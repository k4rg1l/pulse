# Pulse — Unbounded Exploration Research Dump

**Date:** 2026-06-21 · **Baseline:** Pulse v0.4.1 (Windows tray app monitoring OpenRouter) · **Repo:** https://github.com/k4rg1l/pulse

> This is a pure-research artifact. No code was changed; nothing in the app was touched. Every idea below carries a **Feasibility** line (Verified or SPECULATIVE, citing the API/library/Win32 call/Qt class that makes it real), a **Value** line, and an **Effort** estimate sized for *one solo dev with an AI pair*. Findings were gathered by parallel research agents and cross-checked against current (mid-2026) documentation and the actual Pulse source.

## What Pulse actually is (the bones we can build on)

Pulse is a ~3,000-line, nine-module PySide6 app. The architecture is the asset here, so it's worth stating plainly because almost every idea below reuses it:

- **A signal-driven worker thread does *all* I/O.** `APIWorker(QObject)` lives on a `QThread`; `FetchTrigger` signals queue work onto it; `*_ready(object)` signals marshal results back to the main thread, where — and *only* where — widgets are touched. This is exactly the shape every new data source wants: poll off-thread, render on-thread.
- **A frameless, never-focused popup.** `FramelessWindowHint | WindowStaysOnTopHint | BypassWindowManagerHint`. Because it never holds focus, click-outside dismiss polls `GetForegroundWindow` and compares PIDs rather than using `focusOutEvent`. New floating UI must respect the overlay-child vs. top-level-Tool split documented in AGENTS.md.
- **A single-instance kernel mutex** (`CreateMutexW("Global\\Pulse_SingleInstance_v1")`), atomic JSON persistence (`%APPDATA%/Pulse/state.json`, 90-day retention, BOM-tolerant reads), and a `Settings` loader that *silently drops unknown keys* — meaning new config fields are additive and backward-compatible by construction.
- **The Pinned Models section already is a card stack** — a `QVBoxLayout` of dynamically created/removed/reordered `PinnedModelCard` widgets. The plugin/card-stack model isn't a rewrite; it's generalizing a mechanism that already exists.

Two hard constraints frame everything: **honor the Qt + Win32 invariants in AGENTS.md** (no widget access off the main thread, the `WS_EX_TOOLWINDOW`-after-show dance, the overlay/top-level split), and remember the shipping reality — an **unsigned PyInstaller `.exe`** where `sys.stderr is None` and global input hooks trip antivirus.

A note on philosophy versus AGENTS.md: that file explicitly says *"don't add features the user hasn't asked for"* and *"don't promise multi-provider abstraction."* This document is deliberately the opposite — an unbounded map of the possible. The synthesis below reconciles the two by ranking on **strategic fit**, so the items that float to the top are the ones that either *are* already-requested roadmap items or are low-risk, on-identity expansions, while the genuinely speculative material is honestly quarantined into Wildcards, Watching, and Abandoned.

# Top 15 Greenlight Now

Highest confidence × value × strategic fit, each shippable by one dev with an AI pair without fighting the invariants. Ordered roughly by leverage.

### 1. Daily spend tracking
- **Feasibility:** Verified — derived entirely from data Pulse *already* persists (`History.snapshots`, `/api/v1/key` usage). No new external dependency.
- **Value:** The single most on-identity feature possible: "You spent $4.20 today / $61 this month" answers the user's actual question. Already implied by the roadmap's cost-calculator and per-model-spend items.
- **Effort:** hours (a display layer over existing data).
- Compute `balance_at_midnight − balance_now`, special-casing top-up jumps (positive balance deltas) so they aren't misread as negative spend. Zero new I/O, zero new risk. This is the cheapest high-value win in the entire research.

### 2. Alert rules engine + native Windows toasts
- **Feasibility:** Verified — a pure-Python declarative evaluator over the existing worker sample stream, emitting a Qt signal that drives a toast via `win11toast` / `Windows-Toasts` (WinRT). This is roadmap item #1.
- **Value:** The engine behind every notification users actually want — top-up triggered, balance-out, provider outage on a pinned model, GPU thresholds. Declarative rules (`{"metric":"balance","op":"<","threshold":5,"for_minutes":0}`) let users add alerts without code, and the MCP server + webhook receiver can register rules through the same path.
- **Effort:** days (robust evaluator with duration/hysteresis); hours for threshold-only.
- Fire on the *transition* into the alert state with a clear-band to debounce oscillation. Raise the toast from the main thread on the `alert_fired` signal — never from the worker. Register an AppUserModelID so toasts read "Pulse," not "Python."

### 3. Global hotkey to summon the popup
- **Feasibility:** Verified — `RegisterHotKey` via `ctypes.windll.user32` (the channel `_show_no_activate` already uses), running a tiny `PeekMessage` pump on a dedicated thread that emits a `summon_requested` signal. **Qt `QShortcut` will NOT work** — it's focus-scoped and the dashboard never holds focus. Roadmap v0.5 item.
- **Value:** The highest-frequency interaction once the app is trusted; removes the tray-hunt entirely (relevant given the first-run overflow-tray quirk).
- **Effort:** hours to ~1 day (with a configurable binding + conflict handling).
- `RegisterHotKey` returns `False` silently if the combo is taken — check it and surface a toast. Avoids third-party deps and the AV-flagging that the `keyboard` lib's low-level hook attracts.

### 4. NVIDIA GPU stats (NVML)
- **Feasibility:** Verified — `nvidia-ml-py` (`pynvml`): `nvmlDeviceGetUtilizationRates`, `nvmlDeviceGetMemoryInfo`, `nvmlDeviceGetTemperature`, `nvmlDeviceGetPowerUsage`. No admin; reads through the installed driver.
- **Value:** The most on-brand hardware metric for Pulse's actual audience — people watching an AI-spend gauge are frequently NVIDIA-GPU AI/ML users. "GPU 84% · 11.2/12 GB · 71°C · 210 W" is exactly what this crowd loves.
- **Effort:** hours (NVIDIA path).
- `nvmlInit()` once on the worker inside try/except; on failure set a flag and skip GPU polling entirely so it degrades silently on AMD/Intel/no-GPU machines. Poll 2–3s.

### 5. Core system vitals via psutil (CPU / RAM / disk / network)
- **Feasibility:** Verified — `psutil` 7.x: `cpu_percent(interval=None)`, `virtual_memory()`, `disk_usage`, `net_io_counters()`. No admin; bundles cleanly in PyInstaller.
- **Value:** Universal — gives Pulse a reason to be glanced at even when the balance hasn't moved. Network up/down rates pair thematically with Pulse's HTTP-API identity.
- **Effort:** hours.
- Prime `cpu_percent` once at worker startup (first call returns 0); keep `interval=None` so it never blocks the worker. Derive net rates from counter deltas over wall-clock; clamp negative deltas (adapter resets) to zero.

### 6. Battery & power state
- **Feasibility:** Verified — `psutil.sensors_battery()` (wraps `GetSystemPowerStatus`); returns `None` on desktops.
- **Value:** Laptop users keeping a tray app open all day. Cleanly hides itself on desktops.
- **Effort:** hours.
- Poll slowly (30–60s). Handle `POWER_TIME_UNLIMITED`/`POWER_TIME_UNKNOWN` so you never show a garbage ETA in the first minute after a plug/unplug.

### 7. FX rates → show credits in home currency (Frankfurter)
- **Feasibility:** Verified — `GET https://api.frankfurter.dev/v1/latest?base=USD&symbols=INR`. Keyless, no signup, ECB-sourced, self-hostable. (`exchangerate.host` is now key-gated — don't use it.)
- **Value:** Directly on-identity and personally relevant — the user is INR-based; rendering OpenRouter's USD balance also as "≈ ₹1,035" is a low-friction, genuinely useful enhancement.
- **Effort:** hours.
- FX moves slowly — poll once or twice a day and cache (read cached JSON with `utf-8-sig` per the BOM invariant). ECB reference rates, fine for a "≈" display.

### 8. Pomodoro / focus timer
- **Feasibility:** Verified — pure Qt: a main-thread `QTimer` state machine (work/short/long break), the tray icon/tooltip as the countdown, tray notifications on phase change. No dependency, no network.
- **Value:** Universal among the focus crowd; turns a glance-tool into a daily driver and anchors a later Focus Mode (mute audio + suppress new toasts via the verified `NOC_GLOBAL_SETTING_TOASTS_ENABLED` registry toggle).
- **Effort:** hours — the cheapest item here and the best probe of the productivity direction.
- Persist phase + remaining seconds so a crash mid-session resumes.

### 9. Local active-window screen time
- **Feasibility:** Verified — `GetForegroundWindow` → `GetWindowThreadProcessId` → `psutil` exe name (Pulse already uses `GetForegroundWindow` for dismiss). Poll 1–2s on the worker, bucket dwell time per exe.
- **Value:** A fully-local "where did my day go" breakdown with no cloud service — the RescueTime/ActivityWatch use case, privacy-first. Feeds the heat-map and break-reminder wildcards.
- **Effort:** days.
- Store **process-level** aggregates by default; make window-title capture opt-in (titles leak doc names/URLs). Use `GetLastInputInfo` for idle detection so AFK time isn't billed. Persist rollups, not raw event streams.

### 10. GitHub PRs/notifications + Actions CI status
- **Feasibility:** Verified — `GET /notifications` (with `X-Poll-Interval` + conditional requests, ~free on rate limit), `GET /search/issues?q=review-requested:@me`, `GET /repos/{o}/{r}/actions/runs?branch=…&per_page=1`. One PAT (paste, no OAuth dance) or the `gh` CLI.
- **Value:** Very high for *this exact user* — the OpenRouter-monitor author is squarely the target. "2 PRs awaiting review · main: ✓ passing" is a killer dev-tray line.
- **Effort:** hours.
- Use `/notifications` as the primary signal (conditional requests cost no rate-limit budget); reserve the Search API (30 req/min) for explicit refreshes.

### 11. Linear assigned-issue count
- **Feasibility:** Verified — `POST https://api.linear.app/graphql` with a personal API key, `viewer { assignedIssues(filter:{…}) { totalCount } }`.
- **Value:** Strong for the IC-engineer/PM audience Pulse skews toward; "5 issues assigned to you" is a clean tray line. The lowest-friction SaaS integration available — personal key, no OAuth.
- **Effort:** hours.
- Request only `totalCount` to stay cheap; complexity-based rate limit (5,000/hr) is a non-issue at 2–5 min cadence.

### 12. Todoist today-tasks
- **Feasibility:** Verified — `GET https://api.todoist.com/api/v1/tasks/filter?query=today`, bearer token (paste). Target the new `/api/v1/` endpoints (REST v2/Sync deprecating through early 2026).
- **Value:** "3 tasks due today" — glanceable, zero OAuth friction.
- **Effort:** hours.
- Respect `Retry-After` on 429; the ~1000 req/15min limit is irrelevant here.

### 13. MCP server exposing Pulse's data (the keystone)
- **Feasibility:** Verified — the official `mcp` Python SDK (`FastMCP`), **Streamable HTTP transport bound to `127.0.0.1`**, hosted in-process on a dedicated thread (FastMCP/uvicorn runs its own asyncio loop and must NOT share the Qt main thread). Tools read a thread-safe snapshot freely; mutations emit a Qt signal.
- **Value:** The strategic unlock. Lets Claude Code / Claude Desktop answer "what's my OpenRouter balance and burn rate?" or set up "tell me when balance drops below $5" with zero chat UI built by Pulse. The CLI companion and webhook story can be re-expressed on the same seam — so build this first.
- **Effort:** days (read-only server); 1–2 weeks (event subscriptions + polished Claude Desktop config docs).
- Bind localhost only. Keep tools read-mostly initially; `set_alert`/`pin_model` cross into "agent can change my app" and deserve a confirmation toast. The single-instance mutex makes the port owner deterministic.

### 14. Local-LLM daily summary (Ollama)
- **Feasibility:** Verified — `POST http://localhost:11434/api/generate` with `"stream": false`, plain `requests` on the worker thread; text delivered to the toast via signal.
- **Value:** Turns the planned daily-summary toast from a templated string into prose ("Quiet day — $4.12, mostly Claude on the coding agent; burn rate trending down"). Privacy-positive, and the rare AI feature that *reduces* OpenRouter spend.
- **Effort:** hours.
- Assemble the day's numbers deterministically; ask the LLM only to *phrase* them, never to invent figures. Catch `ConnectionError`/404 and fall back to the template so the notification always fires.

### 15. Card / plugin refactor — Phase 1 (hardcoded card list)
- **Feasibility:** Verified — the Pinned Models section already implements the card-stack pattern (`_pinned_layout`, the detach-and-re-add reorder loop); the worker→signal→main flow already matches the proposed `poll()`/`render()` lifecycle. None of the AGENTS.md invariants live in the section-building code.
- **Value:** The architectural leverage that makes *everything else* coherent — notifications, cost calculator, per-model spend, and a settings GUI all become cards with a common lifecycle. Pitch it as enabling the already-requested roadmap, not as speculative expansion.
- **Effort:** ~1 week (Card ABC + host + scheduler ≈ 1.5d, migrate 5 sections ≈ 2d, settings keys ≈ 1d, full 20-point re-validation ≈ 1d).
- Ship Phase 1 (hardcoded `_BUILTIN_CARDS`, no drag) first and validate before layering discovery/reorder. Fan one `/api/v1/key` fetch out to the three cards that consume it rather than triple-fetching.

**Honorable mentions** (greenlight-grade, just outside the 15): microphone/camera in-use indicator (verified HKCU `LastUsedTimeStop == 0`, distinctive privacy signal); weather + air quality via keyless Open-Meteo (AQI is genuinely actionable in India); Prometheus exporter for free Grafana time-series; Vercel/Netlify deploy status; multi-monitor change-awareness (closes a real bug for the 3440-wide user).

# The 10 Wildcards

High-risk, high-reward. Would be amazing if they land; each has a real failure mode.

### 1. Claude Code cost via local `.jsonl` parse
- **Feasibility:** Verified mechanism, SPECULATIVE durability — parse `~/.claude/projects/**/*.jsonl` session logs directly (what TokenTracker-style tools do) for exact local token/cost. No keys, no org, no OTel setup. Risk: undocumented format that can shift between releases.
- **Value:** Huge and personal — exact, free, fully-local Claude Code spend, the user's primary tool. Far lower friction than the OTel push-collector path.
- **Effort:** hours (parse) — the payoff-to-effort ratio is the best of any AI-spend source, hence its wildcard status despite the format risk.

### 2. Cursor Admin API per-member spend
- **Feasibility:** Verified — `POST https://api.cursor.com/teams/spend` (Basic auth, team key as username), real dollar spend per member in cents. Better than most inference providers' own spend APIs.
- **Value:** High for any Cursor *team admin*. Risk/limit: team-only (solo non-team users excluded), ~20 req/min, and Dec-2025 deprecations around per-user caps to mind.
- **Effort:** days (POST-with-body client, Basic auth, one section).

### 3. Keystroke / click / scroll counters + WPM estimate
- **Feasibility:** Verified but fraught — `WH_KEYBOARD_LL`/`WH_MOUSE_LL` global hooks (or `pynput`), **counts only, never key identity**. The hook proc must do nothing but `count += 1` and return immediately.
- **Value:** A delightful quantified-self novelty ("12,400 keystrokes · scrolled ~2.3 km today"). The landmine: a global low-level keyboard hook is structurally a keylogger — AV/EDR will flag the unsigned `.exe`, compounding existing first-run friction.
- **Effort:** days (raw ctypes) / hours (`pynput`).
- Strictly opt-in, counts-only, local-only, with an upfront AV warning. The mouse-travel variant via `GetCursorPos` polling needs **no hook** and dodges the AV risk entirely — prefer it.

### 4. Wi-Fi signal strength / SSID
- **Feasibility:** Verified — `WlanQueryInterface` (`wlanapi.dll`) via ctypes returns `wlanSignalQuality` (0–100), locale-proof. The `netsh` shell-out path is hours but breaks on localized Windows.
- **Value:** A signal bar that drops right as an OpenRouter request times out is a satisfying correlation. Risk: the ctypes struct definitions (`WLAN_CONNECTION_ATTRIBUTES`) and `WlanFreeMemory` cleanup are non-trivial.
- **Effort:** days (robust ctypes path).

### 5. Heat-map-of-your-day
- **Feasibility:** Verified — pure `QPainter` `drawRoundedRect` grid (same technique as `TimelineChart.paintEvent`), fed by `History.snapshots` timestamps + usage deltas.
- **Value:** High and cheap — a GitHub-style day-of-week × hour-of-day spend heatmap answers "when do I actually burn credits?" Nobody else surfaces this for OpenRouter.
- **Effort:** days.
- Render empty buckets as the border track color (idle-dedup makes rows sparse), not as zero-spend.

### 6. Rewind mode
- **Feasibility:** Verified — `History.balance_series`/`topup_events` parameterized to an arbitrary `[t0,t1]` window, fed into `TimelineChart.set_data`, scrubbed by a `QSlider`.
- **Value:** Delightful forensics — "what happened at 3am that drained $4?" Risk is low but value is occasional.
- **Effort:** days.
- Strictly read-only; disable live poll timers while scrubbing so burn-rate math (which assumes monotonic real-time samples) isn't corrupted.

### 7. Snap-to-corner persistent HUD overlay
- **Feasibility:** Verified — `QScreen.availableGeometry()` (already used for clamping) + a settings `anchor_corner` enum; cycle the dashboard to a monitor corner as an always-visible HUD.
- **Value:** Multi-monitor users get a persistent balance HUD instead of a summoned popup. Risk: it *inverts* the dismiss model — pin mode must disable `dismiss_on_focus_loss` and the foreground-polling timer, and re-apply `WS_EX_TOOLWINDOW` after every move.
- **Effort:** days.

### 8. Global command palette (Ctrl+K)
- **Feasibility:** Verified — a frameless `Tool` window + `QLineEdit` + filtered list (reuse the `ModelPicker` fuzzy-filter), summoned by the same global-hotkey mechanism; actions enumerate from existing signals (`refresh_requested`, `toggle_dashboard`, `_open_settings_file`, link URLs).
- **Value:** Power users (the stated audience) jump to any action without hunting the tray. Risk is low; value scales with the number of actions.
- **Effort:** days.
- Must use `WA_ShowWithoutActivating` + the `WS_EX_TOOLWINDOW`-after-show dance or it steals focus and earns a taskbar slot — reuse `_show_no_activate`.

### 9. ActivityWatch ingestion
- **Feasibility:** Verified — ActivityWatch's local REST API (`http://localhost:5600/api/0/buckets/{id}/events`), no auth (localhost-only).
- **Value:** If the user already runs AW (a popular FOSS choice), Pulse gets rich, pre-categorized window/AFK data without maintaining its own hook. Risk: a hard external dependency — must degrade gracefully when absent.
- **Effort:** hours (read events) / days (full query-API aggregations).

### 10. Themeable skins ecosystem
- **Feasibility:** Verified — `theme.py` already centralizes a `Colors`/`Fonts`/`STYLESHEET`; convert `Colors` to a loadable palette (`%APPDATA%/Pulse/themes/*.json`). `QPainter` widgets read `Colors.*` at paint time, so a live swap is just a global `update()`.
- **Value:** Broad appeal and a natural "first community plugin" hook — skins + cards = a real ecosystem. Risk/blocker: scattered inline hex/HTML colors (`SectionHeader`, `ProviderPopup` border `#00d2ff`, tooltip HTML) must first be routed through `Colors` or skins apply half-way.
- **Effort:** days; best done *after* the card refactor so themes and cards ship as one story.

## AI & Developer-Tool Sources

Pulse today is a single-source spend monitor: a worker thread polls OpenRouter, marshals JSON back to the main thread via signals, and renders it in a frameless dark popup. Every source below was evaluated for whether it fits that exact shape — a poll-and-display loop with no OAuth dance, no local agent, and a key the user already has or can mint in two minutes. The honest finding: AI inference providers fall into two camps. The big three (Anthropic, OpenAI) and the dev-tool platforms (Cursor, GitHub Copilot) have shipped real, documented, key-authenticated usage/cost endpoints. The smaller inference shops (Groq, Together, Fireworks, Replicate) mostly have dashboards and *spend limits* but no read-back-your-spend API — a critical asymmetry Pulse must not paper over.

### OpenRouter (current)
- **Feasibility:** Verified — `GET /api/v1/key` (user key, returns `limit`/`usage`/`limit_remaining`), `GET /api/v1/credits`, `GET /api/v1/models`. `/api/v1/activity` and `/api/v1/keys` require an **org-scoped management key**, not a user key (per AGENTS.md).
- **Value:** The existing user; this is the baseline. Anyone routing multi-provider inference through OpenRouter gets near-complete spend visibility from one key.
- **Effort:** Done. Per-model spend via `/activity` is the next increment — days, gated on management-key data accumulating.
- Already integrated. Gotchas live in AGENTS.md: pricing fields are strings (`float()` them), `latency_last_30m` is a `{p50,p75,p90,p99}` dict, and `usage`/`usage_daily`/`usage_weekly`/`usage_monthly` collapse to the same value when all activity is same-day. Polling at 60–300s is fine.

### Anthropic Console / Admin API
- **Feasibility:** Verified — `GET /v1/organizations/cost_report` and `GET /v1/organizations/usage_report/messages`, documented at platform.claude.com. Requires an **Admin API key** (`sk-ant-admin...`), mintable only by an org owner in Console; a normal `sk-ant-api03` key will **not** work.
- **Value:** Huge for the user — they live in Claude. Anyone with a Claude API org gets daily cost broken down by workspace/description and token usage bucketed by model, service tier, and context window.
- **Effort:** Days. New `AnthropicClient` mirroring `api_client.py`; new settings field `anthropic_admin_key`; one dashboard section.
- Cost report groups by `workspace_id`/`description`; usage report buckets at 1m/1h/1d with rich `group_by`. Data lands within ~5 min; poll once per minute max. Gotcha: Priority Tier costs are excluded from the cost endpoint — reconstruct from the usage endpoint or a naive "total spend" undercounts. The Admin-key barrier means solo developers on a Pro/personal plan (not an org) can't use this at all — be honest in the UI.

### OpenAI Usage / Costs API
- **Feasibility:** Verified (current) — `GET /v1/organization/costs` and `GET /v1/organization/usage/completions`, both requiring an **Admin key** (`sk-admin...`). Verified (dead) — the old `/v1/usage` and `/v1/dashboard/billing/*` endpoints now return empty arrays or 404 for normal `sk-` keys; do not build on them.
- **Value:** Large user base. Costs endpoint gives daily spend with `group_by` of `project_id`, `line_item`, `api_key_id`.
- **Effort:** Days. Same client/settings/section pattern as Anthropic.
- Costs `bucket_width` is **1d only**, `limit` 1–180 buckets, times in Unix seconds. Gotchas: (1) Admin-key requirement locks out hobbyists without an org; (2) intermittent 404s on `/v1/organization/costs` logged in the community, so code defensively; (3) data can lag the dashboard. Poll daily-ish.

### AWS Bedrock
- **Feasibility:** Verified but heavy — no single "spend" endpoint. **CloudWatch** `GetMetricData` over `AWS/Bedrock` (`Invocations`, `InputTokenCount`, `OutputTokenCount`, `InvocationLatency`) for usage telemetry, and **Cost Explorer** `GetCostAndUsage` for dollars. Both via boto3, SigV4-signed, needing IAM creds with `cloudwatch:GetMetricData` and `ce:GetCostAndUsage`.
- **Value:** Enterprise/AWS-native teams only. Niche for Pulse's audience.
- **Effort:** Weeks. SigV4 signing, region handling, IAM docs, token→cost translation.
- Cost Explorer data is delayed up to 24h and **costs $0.01 per API request** — naive minute-polling literally bills the user. The IAM-credential UX (access key + secret) is a poor fit for a paste-one-key tray app. Defer.

### Groq
- **Feasibility:** SPECULATIVE — **no public usage/billing read API confirmed.** Groq has a console Usage view and org-wide spend limits, but the widely-cited `/api/v1/usage` endpoint appears in community/marketing text, not the official API reference. To check: console.groq.com/docs/api-reference before building.
- **Value:** Moderate — fast, cheap inference; users would want a balance/spend readout.
- **Effort:** Hours *if* the endpoint exists and takes the `gsk_` key; otherwise impossible without scraping.
- Spend tracking has a documented 10–15 min delay. Until the endpoint is verified, don't ship a Groq card.

### Together AI
- **Feasibility:** SPECULATIVE/negative — docs cover billing *limits* and the dashboard, but no documented programmatic **read** of spend/balance. The API is OpenAI-compatible (inference only).
- **Value:** Moderate; open-model users.
- **Effort:** Blocked until an endpoint is confirmed.
- You can read `/v1/models` (a catalog, not spend). Treat as no-API for Pulse's purpose.

### Fireworks AI
- **Feasibility:** SPECULATIVE — OpenAI-compatible inference is documented; a dollar-denominated usage/billing read API is not surfaced. Per-token billing lives in the dashboard.
- **Value:** Moderate; serverless open-model crowd.
- **Effort:** Blocked pending verification at docs.fireworks.ai.
- Same story as Together. No card until proven.

### Replicate
- **Feasibility:** SPECULATIVE for *billing*, Verified for *activity-by-proxy*. `GET /v1/account`, `GET /v1/predictions` are real and `r8_`-key auth'd, but **billing/credit-balance is not exposed**. Since July 2025, new accounts are prepaid-credit; "work stops at zero" yet no documented endpoint reads that balance.
- **Value:** Moderate; image/video users who'd love a "credits remaining" gauge — exactly what's unavailable.
- **Effort:** Days for a usage-events approximation via `/v1/predictions` (sum hardware-seconds × price); the headline "balance" is blocked.
- You could *estimate* spend by walking predictions and applying per-second rates, but it's fragile and diverges from the real invoice. Pulse can show *activity*, not *balance*.

### Cursor
- **Feasibility:** Verified — **Admin API** at `https://api.cursor.com`, **Basic auth** with the team key as username. `POST /teams/spend` (current-cycle spend per member, in cents), `POST /teams/daily-usage-data`, `POST /teams/filtered-usage-events`, `/teams/members`.
- **Value:** High for any Cursor team admin — real dollar spend per member. A *better* spend API than most inference providers ship.
- **Effort:** Days. POST-with-JSON-body client, Basic-auth header, one section.
- Rate limit ~20 req/min per team. **Team-only**: only admins can mint keys, so solo non-team users are excluded. Mind Dec-2025 changes (per-user spend caps deprecated for alerts; Enterprise pricing groups). Community MCP `ofershap/cursor-usage` wraps the same API as a reference.

### Claude Code metrics (OTel)
- **Feasibility:** Verified, but **push-not-pull**. Claude Code emits OpenTelemetry when `CLAUDE_CODE_ENABLE_TELEMETRY=1` + an OTLP exporter is set — token counters, total cost, session count as metrics. There is **no endpoint to GET**; Pulse would have to *be* an OTLP collector.
- **Value:** High personally — local, free, exact Claude Code cost. No org or admin key.
- **Effort:** Weeks for the OTLP-receiver path. The pragmatic alternative — parse `~/.claude/projects/**/*.jsonl` session logs directly — is hours, fully local, no env-var setup. That's the Pulse move (and the wildcard above).
- Default metric export interval 60s.

### Codex / OpenAI Codex CLI
- **Feasibility:** SPECULATIVE — **no public usage API and no clean JSON export today.** Limits show only via the interactive `/status` command; `codex usage --json` is an open feature request (openai/codex#15281), not shipped.
- **Value:** Moderate for Codex users — but unreachable cleanly.
- **Effort:** Blocked. A local-session-file parse is theoretically possible but the format is undocumented; high risk, low payoff.
- Wait for `codex usage --json` to land, then revisit.

### GitHub Copilot
- **Feasibility:** Verified — REST Copilot Metrics API. `GET /orgs/{org}/copilot/metrics`, `GET /enterprises/{enterprise}/copilot/metrics`, plus newer report endpoints returning signed NDJSON URLs. Auth via PAT/OAuth with `manage_billing:copilot` or `read:enterprise`.
- **Value:** High for org/enterprise admins — adoption, acceptance rates, active users per feature.
- **Effort:** Days. Standard bearer-token GET; report endpoints add an NDJSON-download step.
- This is **engagement/adoption metrics, not dollar spend**. Org-admin scope required, so individual subscribers can't use it. Good for a "team adoption" panel, wrong tool if the user wants cost.

### Cline / other coding agents
- **Feasibility:** SPECULATIVE for any *pull* API — Cline shows token count + estimated cost in its own UI and ships telemetry to PostHog (push, opt-in), no public read endpoint for individuals. Local angle: Cline writes task/history data on disk that third-party local trackers parse.
- **Value:** Moderate; growing open-source agent crowd.
- **Effort:** Days for a local log-file parse; blocked for a hosted API.
- The realistic pattern across *all* local coding agents (Claude Code, Cline, Codex, Roo) is one "local agents" module that reads on-disk session logs — not N web integrations. That generalizes far better than chasing per-vendor APIs that mostly don't exist.

**Greenlight candidates:** Anthropic Admin API (verified cost/usage, the user's primary tool); Cursor Admin API (real per-member dollar spend); OpenAI Costs API (verified, mirrors the OpenRouter pattern).
**Wildcard candidates:** Claude Code via local `.jsonl` parse (local, exact, free, hours of work); GitHub Copilot Metrics (verified/rich but adoption-not-cost); Replicate activity estimation (only way in since balance has no API).
**Abandoned:** AWS Bedrock (per-call billing + IAM); Groq/Together/Fireworks (no verified spend API, dashboards only); Codex CLI (no usage API); OpenAI legacy `/v1/usage` & billing endpoints (dead for normal keys).

## System & Hardware Monitors

What is *actually* reachable from Python on Windows 11 in 2026 via ctypes, WMI, `psutil`, or vendor SDKs. Pulse's architecture forces a hard constraint on every item: all polling and I/O run on the worker thread, results marshaled to the UI through signals. Most candidates are a natural fit (poll-and-emit metrics), with two exceptions (USB and brightness-change events) that are *push* events needing a hidden message window or a polling fallback. A recurring theme: the friendly metrics (CPU, RAM, battery, Wi-Fi quality, NVIDIA GPU) are clean and cheap; the "sexy" metrics (temps, fan RPM, AMD/Intel GPU, SMART) almost all require elevation or a third-party kernel driver — a poor fit for a lightweight tray companion shipped as an unsigned PyInstaller exe.

### CPU / RAM / Disk / Network via psutil
- **Feasibility:** Verified — `psutil` 7.x: `cpu_percent(interval=None)`, `virtual_memory()`, `disk_usage(path)`, `disk_io_counters()`, `net_io_counters()`. No admin. Bundles cleanly in PyInstaller.
- **Value:** Universal. A compact "system load" strip gives Pulse a reason to be glanced at even when the balance hasn't moved.
- **Effort:** hours.
- Call `cpu_percent(interval=None)` (non-blocking) per poll; prime it once at worker startup. Never pass `interval>0` on the worker — it blocks that thread. Emit a dict via signal. Poll 2–3s.

### Network up/down rates
- **Feasibility:** Verified — derived from `psutil.net_io_counters()` cumulative bytes; rate = delta ÷ delta-wall-clock. `pernic=True` to ignore virtual adapters. No admin.
- **Value:** A live "↓ 4.2 MB/s ↑ 0.3 MB/s" readout, useful on metered/flaky connections and thematically aligned with Pulse's network identity.
- **Effort:** hours.
- Store previous counter + timestamp; clamp negative deltas (adapter reset) to zero. 1–2s cadence.

### Per-process CPU
- **Feasibility:** Verified — `psutil.process_iter([...])`; first reading per process is 0.0 (needs a baseline). Catch `AccessDenied` on system processes.
- **Value:** A "top 3 CPU hogs" mini-list answers "why is my fan loud?" Niche but appreciated by the power-user audience.
- **Effort:** days — iterating all processes twice, sorting, handling mid-iteration errors; full enumeration is the heaviest poll here.
- Poll no faster than 3–5s. Values >100% (multi-core) are normal; divide by `cpu_count()` for a 0–100 scale.

### Battery % and power state
- **Feasibility:** Verified — `psutil.sensors_battery()` (wraps `GetSystemPowerStatus`); `None` on desktops.
- **Value:** Laptop users. A battery chip + "plugged / N min left" line makes Pulse a credible always-on widget.
- **Effort:** hours.
- Hide the UI entirely when `None`. Handle `POWER_TIME_UNLIMITED`/`UNKNOWN`. Poll 30–60s.

### Wi-Fi signal strength / SSID
- **Feasibility:** Verified (two tiers). Easy: parse `netsh wlan show interfaces` (no admin, but locale-dependent labels). Robust: `WlanOpenHandle` → `WlanEnumInterfaces` → `WlanQueryInterface` from `wlanapi.dll` via ctypes → `wlanSignalQuality` (0–100). No admin.
- **Value:** Anyone on Wi-Fi, especially flaky-connection debugging — a signal bar dropping as an OpenRouter request times out is a satisfying correlation.
- **Effort:** days — the ctypes struct definitions and `WlanFreeMemory` cleanup are non-trivial; `netsh` path is hours but fragile.
- Prefer ctypes (locale-proof). `netsh` parsing breaks on a German/Japanese Windows. Hide on Ethernet-only machines. Poll 5–10s.

### GPU — NVIDIA (NVML)
- **Feasibility:** Verified — `nvidia-ml-py` (`pynvml`): `nvmlDeviceGetUtilizationRates`, `nvmlDeviceGetMemoryInfo`, `nvmlDeviceGetTemperature`, `nvmlDeviceGetPowerUsage`. No admin; reads through the driver.
- **Value:** Huge for Pulse's actual audience — AI/ML users with NVIDIA GPUs. The single most on-brand hardware metric.
- **Effort:** hours (NVIDIA-only path).
- `nvmlInit()` once inside try/except; flag and skip on failure so it degrades silently. Convert mW→W and bytes→GB. `nvmlShutdown()` at exit. Poll 2–3s.

### GPU — AMD and Intel
- **Feasibility:** SPECULATIVE. AMD: `pyadl` (deprecated ADL) or the modern ADLX SDK with `ADLXPybind` (native build, awkward in PyInstaller). Intel: **no NVML equivalent** on Windows from Python in 2026.
- **Value:** Would broaden coverage, but effort-to-reach is poor and Intel users still see nothing.
- **Effort:** weeks — per-vendor SDK integration with Intel unsolved.
- Promising "universal GPU stats" would be dishonest; abstract behind a provider that tries NVML, then AMD, then gives up.

### CPU/GPU temps & fan RPM (LibreHardwareMonitor / WMI)
- **Feasibility:** SPECULATIVE — not exposed by plain WMI/psutil. Requires **LibreHardwareMonitor running (typically elevated)**, exposing WMI namespace `root\LibreHardwareMonitor` with a `Sensor` class; `WinTmp` wraps it. Its kernel driver needs admin.
- **Value:** Enthusiasts love temps/fan RPM, but it demands a separate elevated app — antithetical to a self-contained tray companion.
- **Effort:** days for the query, but with an unacceptable runtime prerequisite.
- Dark for the vast majority of users; bundling LHM would bloat and complicate signing.

### Screen brightness
- **Feasibility:** Verified for laptops only. WMI `root\wmi`: `WmiMonitorBrightness` (read) + `WmiSetBrightness` (set); `screen-brightness-control` wraps it. **Internal panel only** — most external monitors need DDC/CI.
- **Value:** Modest QoL touch for laptop users; only tangential to Pulse's mission.
- **Effort:** hours (with the library).
- Marshal the *set* request to the worker (WMI = I/O). Hide on external-monitor desktops. Poll on demand / 30–60s.

### USB device insert/remove events
- **Feasibility:** Verified, awkward. Push: `WM_DEVICECHANGE` via `RegisterDeviceNotification`, delivered to a window with a message loop → hook via `QAbstractNativeEventFilter` on the **main thread**. Poll: diff `psutil.disk_partitions()` or WMI `Win32_PnPEntity` on the worker. No admin.
- **Value:** Minor for Pulse — "USB connected" toasts are off-mission.
- **Effort:** days (native filter) or hours (crude polling).
- The native path needs a real HWND; the frameless dashboard may be hidden, so a dedicated message-only window is cleaner. Simplest worker-only approach: diff `disk_partitions(all=False)` every 3–5s.

### Bluetooth device state
- **Feasibility:** SPECULATIVE for "what's connected now." `bleak` is great for scan/connect but not a clean connected-paired-device list. WMI `Win32_PnPEntity` filtered to Bluetooth, or WinRT `Windows.Devices.Bluetooth`, is workable but verbose; "connected vs paired" is inconsistent across stacks.
- **Value:** Low for Pulse.
- **Effort:** days — reliable connected-state across Classic + LE is finicky.
- High effort, low product relevance.

### Audio output/input device & volume
- **Feasibility:** Verified — `pycaw` wraps Core Audio. `AudioUtilities.GetSpeakers()` → `IAudioEndpointVolume` (`GetMasterVolumeLevelScalar`, `GetMute`) + device friendly name. Windows-only, no admin.
- **Value:** Modest standalone; more interesting paired with the mic-in-use indicator.
- **Effort:** hours (volume/mute/name); days for per-app session mixing.
- **Initialize COM on the worker** (`CoInitialize`) before calling. Re-query `GetSpeakers()` on default-device changes rather than caching. Poll 2–5s or on dashboard open.

### Microphone-in-use / camera-in-use indicator
- **Feasibility:** Verified — registry `HKCU\...\CapabilityAccessManager\ConsentStore\microphone` (and `\webcam`), including `NonPackaged`. **`LastUsedTimeStop == 0` means in use now**; nonzero means finished. No admin. Same mechanism as the Windows privacy indicator.
- **Value:** Genuinely on-brand for a privacy-conscious power user — a "🎤 in use by Zoom" / "📷 camera active" tray indicator makes a companion app feel smart.
- **Effort:** hours — registry walk + FILETIME parsing.
- Decode `#`-encoded packaged-app subkey names for friendly labels. Poll 3–5s. A heuristic (the one Microsoft uses), not a kernel guarantee.

### Disk SMART health
- **Feasibility:** SPECULATIVE — needs elevation or a bundled tool. Light path: WMI `root\wmi` `MSStorageDriver_FailurePredictStatus` (`PredictFailure` bool), unreliable across controllers. Full SMART: `pySMART` (wraps `smartctl`), separate install on PATH, admin strongly recommended; NVMe/RAID complicate it.
- **Value:** "Disk health OK/failing" is reassuring but rarely actionable; elevation/bundling cost is high.
- **Effort:** days (WMI bool) to weeks (bundle smartmontools).
- The WMI bool often reads False/unavailable until a drive is genuinely dying (false confidence). Clashes with Pulse's no-elevation shipping model.

**Greenlight candidates:** CPU/RAM/disk/network (psutil); battery state; NVIDIA GPU (NVML); network up/down rates; microphone/camera in-use indicator.
**Wildcard candidates:** Wi-Fi signal (ctypes `WlanQueryInterface`); audio device & volume (`pycaw`, COM-on-worker footgun); screen brightness (laptop-only); per-process CPU (heaviest poll); USB events (off-mission, needs main-thread native filter).
**Abandoned:** AMD/Intel GPU (asymmetric/impossible — no Intel Python path on Windows); temps & fan RPM (needs elevated LibreHardwareMonitor); disk SMART (admin + external binary); Bluetooth state (finicky, low value).

## Productivity & Work-State

Every feature here reuses Pulse's spine (frameless popup, worker thread, ctypes, JSON). The honesty theme: **the Win32 features are cheap and verified; the SaaS features are gated by OAuth friction, not by code.** A single Google or Azure app registration, consent screen, and token-refresh loop is most of the work for half these items. Treat OAuth tokens as persisted state, ideally DPAPI-wrapped (`win32crypt`) or via `keyring`.

### Active-Window History & Time-Per-App Tracking
- **Feasibility:** Verified — `GetForegroundWindow` → `GetWindowTextW` + `GetWindowThreadProcessId` → `psutil` exe name. Event-driven: `SetWinEventHook(EVENT_SYSTEM_FOREGROUND, …, WINEVENT_OUTOFCONTEXT)`.
- **Value:** A local "where did my day go" breakdown (RescueTime/ActivityWatch use case) with no cloud. High value for a privacy-conscious app; the data never leaves the machine.
- **Effort:** days. Polling version (~1s tick on the worker) is half a day; the `SetWinEventHook` version needs a thread with a message loop and the `WINEVENTPROC` ctypes trampoline kept alive in a module-level ref or the process crashes — add a day.
- Append `{exe, title, start_ts, duration_s}`; aggregate per-exe per-day; persist a rolling window. Titles leak doc names/URLs — store process-only by default, title capture opt-in. `GetLastInputInfo` for idle so AFK isn't billed.

### Pomodoro Timer / Focus Sessions
- **Feasibility:** Verified — pure Qt. A main-thread `QTimer` state machine; tray icon/tooltip as countdown; `showMessage()` on phase change. No dependency.
- **Value:** Universal among the focus crowd; pairs with time-tracking (a Pomodoro auto-tags the active session). The cheapest item here and the best probe of the productivity direction.
- **Effort:** hours.
- Persist phase/remaining/count so a crash mid-session resumes. Integrate with Focus Mode so starting a Pomodoro optionally flips DND.

### Focus Mode (Suppress Notifications / DND + Mute Audio)
- **Feasibility:** Partially Verified. **Mute audio** Verified via `pycaw` (`SetMute`). **Global toast switch** Verified via registry `HKCU\...\Notifications\Settings\NOC_GLOBAL_SETTING_TOASTS_ENABLED` (DWORD). **But the Windows 11 DND/Focus-Assist profile has no supported toggle API** — the simple registry/`SystemParametersInfo` path was removed at Win10 1803; the real profile relies on undocumented WNF APIs that break without notice.
- **Value:** The payoff for the Pomodoro crowd. Mute + killing new toasts covers ~90% of the felt benefit even without the official DND profile.
- **Effort:** days. `pycaw` mute + toast-toggle is half a day; chasing the real DND profile is a multi-day rabbit hole to abandon.
- Persist pre-focus state and restore on exit/crash. Ship "mute + suppress new toasts," call it Focus Mode, don't promise it drives the OS panel.

### Calendar: Today / Next Event (Google Calendar)
- **Feasibility:** Verified — `GET .../calendar/v3/calendars/primary/events?timeMin=…&singleEvents=true&orderBy=startTime`. Scope `calendar.readonly`; installed-app OAuth (loopback redirect) via `google-auth-oauthlib`.
- **Value:** "Next meeting in 12 min" is one of the highest-utility glance lines possible.
- **Effort:** days, dominated by OAuth. Register a Cloud project + consent screen; **unverified apps stay in "testing" with 7-day refresh-token expiry** — a real friction wall for a hobby app. Code is half a day once a token exists.
- All-day events use `start.date`, not `start.dateTime` — handle both. `singleEvents=true` is mandatory or recurring events return master rows. Poll 5–10 min.

### Calendar: Today / Next Event (Microsoft Graph / Outlook)
- **Feasibility:** Verified — `GET https://graph.microsoft.com/v1.0/me/calendarView?...`. Delegated `Calendars.Read` + `offline_access`; Azure app registration with MSAL.
- **Value:** The M365/corporate-Outlook population, which often can't use Google. Offer as the alternate backend.
- **Effort:** days. Corporate tenants frequently require admin consent — a hard blocker the user can't self-serve past.
- Want **delegated** permissions; request `offline_access` explicitly or you get no refresh token. Poll 5–10 min.

### Unread Counts: Gmail
- **Feasibility:** Verified — `GET .../gmail/v1/users/me/labels/UNREAD` returns `messagesUnread` directly. Scope `gmail.readonly` / `gmail.metadata`.
- **Value:** "7 unread" in the tray. Reuses the exact Google OAuth project as Calendar — nearly free once one Google feature is authorized.
- **Effort:** days for the first Google feature (OAuth), hours to add on top.
- Use the labels endpoint (don't page messages just to count). Offer a query filter (`is:unread in:inbox category:primary`) if the raw count is noisy. Same 7-day-token verification wall.

### Unread Counts: Slack
- **Feasibility:** SPECULATIVE — **no supported public method returns a global unread/mention count.** `conversations.info` exposes DM-only `unread_count`; `conversations.list` is deprecated. The total badge comes from the **undocumented internal `client.counts` requiring an `xoxc` session token + cookie** — fragile, ToS-adjacent.
- **Value:** High if it worked; the only path is the unsupported one.
- **Effort:** weeks to fan out `conversations.info` per channel — and even that misses channel unreads.
- Honest verdict: a real Slack badge isn't achievable with supported APIs. Abandon unless you accept the `xoxc` hack and warn the user.

### Unread Counts: Linear
- **Feasibility:** Verified — `POST https://api.linear.app/graphql` with a personal API key. `viewer { assignedIssues(filter:{…}) { totalCount } }`.
- **Value:** Strong for IC engineers/PMs — and Pulse's audience skews exactly toward Linear users. The lowest-friction SaaS integration here (personal key, no OAuth).
- **Effort:** hours.
- Request only `totalCount`. Complexity-based limit (5,000/hr) trivially satisfied at 2–5 min cadence.

### Unread Counts: Notion
- **Feasibility:** SPECULATIVE for "unread" — no notification/unread endpoint. `POST /v1/search` or `/v1/databases/{id}/query` (integration token) counts pages matching a filter — a *task* count, not unread.
- **Value:** Modest; overlaps the Task-Lists section.
- **Effort:** hours (token paste, one query) — but only a filtered page count.
- **3 req/s, all tiers**; respect `Retry-After`. Reframe as "Notion task count," not "unread."

### Unread Counts: Discord
- **Feasibility:** SPECULATIVE / discouraged — the Bot API has no per-user unread count; reading a user's unread state requires a self-bot token, which **violates Discord ToS** and risks account termination.
- **Value:** Low for this audience; not worth the exposure.
- **Effort:** N/A — don't build.
- The one integration with account-ban risk. Abandon outright.

### Task Lists: Todoist
- **Feasibility:** Verified — `GET https://api.todoist.com/api/v1/tasks` and `.../tasks/filter?query=today`. Bearer token (paste).
- **Value:** "3 tasks due today" — clean, glanceable, zero OAuth friction.
- **Effort:** hours.
- Target `/api/v1/` (REST v2 + Sync deprecating through early 2026). ~1000 req/15min limit is a non-issue; respect `Retry-After`.

### Task Lists: Microsoft To Do
- **Feasibility:** Verified — `GET https://graph.microsoft.com/v1.0/me/todo/lists` then `.../tasks`. Delegated `Tasks.Read`. Same Azure/MSAL flow as Outlook.
- **Value:** Solid for M365 users; shares the Azure registration with the Outlook calendar feature.
- **Effort:** days for the first MS feature, hours thereafter.
- Same `offline_access` + corporate-tenant admin-consent wall as Outlook.

### Task Lists: Things
- **Feasibility:** Verified-as-impossible — **macOS/iOS only**, AppleScript/URL-scheme automation, no Windows client or web API.
- **Value:** Zero on Windows.
- **Effort:** N/A. Abandon for a Windows target.

### GitHub PR / Issue Status
- **Feasibility:** Verified — `GET /notifications` (with `X-Poll-Interval` + conditional requests), `GET /search/issues?q=is:open+is:pr+review-requested:@me`, assignee search. Fine-grained or classic PAT (paste) or the `gh` CLI.
- **Value:** Very high for this exact user. "2 PRs awaiting review, 1 review requested" is a killer dev-tray line.
- **Effort:** hours to a day.
- Honor `X-Poll-Interval` (~60s); 304 with `If-Modified-Since` costs no rate-limit budget. Search API is only 30 req/min — use `/notifications` as the primary signal.

### CI Build Status (GitHub Actions)
- **Feasibility:** Verified — `GET /repos/{o}/{r}/actions/runs?branch=…&per_page=1` → `status` + `conclusion`. PAT with `repo`/`actions:read`.
- **Value:** "main: ✓ passing" / "✗ failing" right after a push. Reuses the same GitHub PAT — near-free.
- **Effort:** hours on top of the PAT.
- Poll 30–60s while active, back off when idle. Let the user name the workflow file (matrix/multi-workflow case).

### CI Build Status (CircleCI / others)
- **Feasibility:** Verified — CircleCI v2 `GET /pipeline?org-slug=…` → `/workflow`, `Circle-Token` header. Comparable for GitLab CI, Travis.
- **Value:** Niche vs GitHub Actions, which dominates this audience.
- **Effort:** days per additional provider — low value-per-hour after GHA.
- Per-provider token management multiplies settings complexity; gate behind real demand.

### Deploy Status: Vercel
- **Feasibility:** Verified — `GET https://api.vercel.com/v6/deployments?limit=1&projectId=…`, Bearer token, `teamId` if team-scoped.
- **Value:** "Prod: Ready / Building… / Error" for the Vercel crowd. Good for indie devs.
- **Effort:** hours.
- **Team-scoped projects require `teamId`** or you get personal-scope-only results. Tokens long-lived.

### Deploy Status: Netlify
- **Feasibility:** Verified — `GET https://api.netlify.com/api/v1/sites/{site_id}/deploys?per_page=1`, Bearer PAT.
- **Value:** Same use case as Vercel for Netlify users. Offer both behind one provider picker.
- **Effort:** hours.
- Map a friendly site name to its `site_id` (one `GET /sites` at setup). Poll 30–60s active.

### Deploy Status: Fly.io
- **Feasibility:** Verified — Machines API `GET https://api.machines.dev/v1/apps/{app}/machines` → `state`, Bearer token; or shell out to `flyctl status --json`.
- **Value:** Niche but real; the CLI shell-out is attractive since many Fly users already have `flyctl` authenticated.
- **Effort:** hours (CLI) to a day (raw API).
- "Deploy status" is really "are all machines on the latest release and started" — may need `GET /apps/{app}/releases` too. Poll 1–2 min.

**Greenlight candidates:** Pomodoro/focus timer; active-window time tracking (polled); GitHub PRs/notifications + Actions CI (single PAT); Linear assigned-issue count; Todoist today-tasks; Focus Mode (mute + suppress new toasts); Vercel/Netlify deploy status.
**Wildcard candidates:** Google Calendar + Gmail (high value, the Google-OAuth tax + 7-day token expiry); Microsoft Graph (Azure registration + corporate admin-consent walls); Fly.io (`flyctl status --json` shell-out); Notion "task count"; `SetWinEventHook` event-driven tracking (trampoline-crash risk); CircleCI/other CI (only on demand).
**Abandoned:** Slack unread badge (no supported global-count API; only the `xoxc` hack); Discord personal unread (self-bot token = ToS ban risk); Things (macOS/iOS only); true Focus Assist/DND toggle (undocumented WNF, breaks without notice).

## Finance & Personal/Ambient

Pulse's architecture demands a plain HTTP GET returning JSON, pollable on a timer, with no interactive browser flow on the UI thread. Sources needing OAuth redirects, a hosted callback, or a browser-embedded link widget are architecturally hostile to a frameless tray popup. A second filter is *coherence*: the finance sources that fit Pulse's identity (FX to read credits in home currency, an ambient crypto/stock strip) differ sharply in value from a full bank integration, which is a different product.

### Crypto prices (CoinGecko)
- **Feasibility:** Verified — `GET .../api/v3/simple/price?ids=bitcoin,ethereum&vs_currencies=usd`. Keyless public endpoint is throttled/unstable; the free **Demo** plan (`x-cg-demo-api-key`) gives stable **100 calls/min, 10,000/month**, no card.
- **Value:** Crypto holders wanting a glanceable BTC/ETH price without an exchange tab. Marginal-but-pleasant ambient data.
- **Effort:** hours.
- 60s polling (~43k/month) **blows the Demo cap**; 5-min polling (~8.6k/month) stays safe. CoinGecko has *kept* its free tier through 2026.

### Crypto prices (Coinbase / Binance / Kraken public tickers)
- **Feasibility:** Verified — keyless: Coinbase `.../v2/prices/BTC-USD/spot`; Binance `/api/v3/ticker/price?symbol=BTCUSDT`; Kraken `/0/public/Ticker?pair=XBTUSD`. No key/registration for read-only market data.
- **Value:** Same audience; advantage is **no key at all** and per-second freshness.
- **Effort:** hours (one exchange) to a day (abstracting all three — don't over-build).
- Coinbase's spot endpoint is the cleanest shape and safest default; Binance is occasionally geo-restricted (451). Poll 60–300s.

### Stock tickers / indices
- **Feasibility:** Mixed. **Finnhub** Verified, best free tier (`/quote?symbol=AAPL&token=KEY`, 60 calls/min). **Twelve Data** 800/day, 8/min, delayed. **Alpha Vantage** gutted to **25/day** — a trap. **Polygon** 5/min, delayed. **Yahoo (yfinance)** SPECULATIVE — unofficial, frequent 429 IP blocks, unsafe to ship.
- **Value:** Users tracking a few tickers/indices. Lower coherence with Pulse than crypto/FX.
- **Effort:** hours (Finnhub).
- Use **Finnhub** (flat `{"c","d","dp"}` shape). Add a `finnhub_api_key` field mirroring `management_api_key`. Avoid Alpha Vantage and Yahoo unofficial.

### Bank balance / transactions (Plaid)
- **Feasibility:** Verified, but **heavy and ill-fitting**. `POST /accounts/balance/get` requires the full Plaid Link flow (a hosted/JS widget; many banks need an OAuth redirect to their site), then exchange a `public_token`. Live balance is billed **per call (~$0.05–0.15)**; production carries monthly minimums.
- **Value:** High for a net-worth product — but that is **not Pulse**; audience overlap with "OpenRouter credit watcher" is thin.
- **Effort:** weeks (+ ongoing cost + compliance).
- Plaid Link is a browser/OAuth-redirect flow that can't live in the BypassWindowManager popup and has no desktop callback. Per-call billing kills the poll-on-timer model. **Abandon for Pulse.**

### Daily spend tracking
- **Feasibility:** Verified (internal, no external API). Derive from existing snapshot/usage data Pulse already stores.
- **Value:** High and **perfectly on-identity** — literally what Pulse is for. "Spent $4.20 today / $61 this month."
- **Effort:** hours (display layer).
- Compute `balance_at_midnight − balance_now`; special-case top-up jumps so they aren't misread as negative spend. The single highest-value candidate in this section.

### Currency / FX rates
- **Feasibility:** Mixed; two winners. **Frankfurter** Verified, best (`https://api.frankfurter.dev/v1/latest?base=USD&symbols=INR`, keyless, ECB, self-hostable). **open.er-api.com** keyless, attribution required. **exchangerate.host** REGRESSION — **now requires a key** (was keyless).
- **Value:** Directly on-identity. The user is INR-based; rendering USD credits also in home currency ("$12.40 ≈ ₹1,035") is a useful, low-friction enhancement.
- **Effort:** hours.
- Use **Frankfurter**; flat `{"rates":{"INR":83.2}}`. Poll 1–2×/day, cache (read with `utf-8-sig`). ECB reference rates — fine for a "≈" display.

### Electricity rates / spot pricing
- **Feasibility:** Verified **but geo-locked**. **Octopus Energy (UK)** keyless public unit-rates endpoint. **Nord Pool** official feed is commercial. **US utilities** fragmented, no free national API.
- **Value:** High **only** for UK users on a dynamic/Agile tariff who load-shift. Near-zero elsewhere.
- **Effort:** hours (Octopus, UK only); weeks (multi-region).
- Location-dependent opt-in for one market. Octopus shape is clean; poll twice daily after the ~16:00 publish. A niche wildcard, not a default.

### Weather (Open-Meteo, OpenWeatherMap)
- **Feasibility:** Verified. **Open-Meteo** best (`/v1/forecast?...&current=temperature_2m,weather_code`, **no key**, free non-commercial, <10k/day). **OpenWeatherMap** free tier 60/min, 1M/month, but One Call 3.0 needs a card.
- **Value:** Pure ambient nicety — a small temp/icon in the tooltip/footer. The form factor suits it.
- **Effort:** hours.
- Use **Open-Meteo** (keyless is decisive). Needs lat/long — ask once or geolocate by IP. Poll 15–30 min. Free tier confirmed intact in 2026.

### Sunrise/sunset
- **Feasibility:** Verified, two routes. **API** `api.sunrise-sunset.org/json?lat=..&lng=..` (keyless). **`astral`** Python lib — zero network, computes offline.
- **Value:** Minor ambient flourish (auto-dark after sunset, "🌇 18:42").
- **Effort:** hours.
- **Prefer `astral`** — no network dependency, never rate-limits or fails offline. Compute once/day. Shares the stored lat/long.

### Air quality (Open-Meteo Air Quality, AQICN)
- **Feasibility:** Verified. **Open-Meteo Air Quality** best (`air-quality-api.open-meteo.com/v1/air-quality?...&current=european_aqi,pm2_5`, no key). **AQICN** needs a free token.
- **Value:** Relevant in high-pollution regions — **India specifically** has frequent severe AQI events, plausibly on-point for this user. A "AQI 168 ⚠️" chip is glanceable and actionable.
- **Effort:** hours.
- Use **Open-Meteo Air Quality** — keyless, shares the weather integration's lat/long and HTTP pattern, so it's nearly free to add alongside weather. Poll 30–60 min.

**Greenlight candidates:** Daily spend tracking (highest value, zero new deps); FX via Frankfurter (keyless, on-identity INR display); crypto via Coinbase ticker or CoinGecko Demo; weather + air quality via Open-Meteo (keyless, one lat/long, ship together).
**Wildcard candidates:** Stock tickers via Finnhub (best free tier, but one step from Pulse's identity); sunrise/sunset via `astral` (trivial, cosmetic); Octopus Agile electricity (high value but UK-only, opt-in).
**Abandoned:** Plaid (architecturally hostile, per-call billing, different product); exchangerate.host (now key-gated); Alpha Vantage (25/day); Yahoo unofficial (fragile 429s); Nord Pool / US utility electricity (commercial / no free national API).

## Life Telemetry / Quantified Self

The governing principle is **privacy-first, local-only**: telemetry that originates on the machine (window titles, input counts, water logs) never leaves it; the only network calls are OAuth exchanges and read-only GETs against APIs the user explicitly authorized. There is no Pulse backend. For input-derived features that posture is the difference between a tasteful self-metric and something AV (or the user's own conscience) treats as spyware. A subtle wrinkle: a global `WH_KEYBOARD_LL`/`WH_MOUSE_LL` hook needs a message pump on its owning thread (Pulse's main Qt loop qualifies) and the callback must be near-instant — just increment a counter; aggregate on a timer, never in the hook.

### Fitbit Web API (steps / activity)
- **Feasibility:** Verified — `GET /1/user/-/activities/date/{date}.json` (`activity` scope); intraday under the `intraday` scope. **Authorization Code Grant with PKCE**; 8-hour access tokens, rotating refresh tokens. A developer's **own** intraday data is auto-available to a "Personal" app with no special approval.
- **Value:** The classic step/active-minutes line. A Pulse owner registering their own Personal app gets minute-level steps free.
- **Effort:** days. The OAuth2 PKCE dance (localhost redirect listener, system browser, code exchange, persist refresh token) is the bulk; data GETs are trivial.
- Store only the refresh token; refresh on the worker on a 401. Poll daily totals every 15–30 min; intraday on dashboard-open. Local-only.

### Google Fit / Health Connect (steps / activity)
- **Feasibility:** SPECULATIVE, dead-end on Windows. **Google Fit REST is deprecated** (sign-ups closed May 2024, winding down through 2026). **Health Connect is Android-only on-device** with no REST/cloud surface. Google routes cross-platform integrations to Fitbit (which it owns) or Apple HealthKit.
- **Value:** Would have served Android users without a Fitbit, but the platform story collapsed for desktop.
- **Effort:** weeks, almost certainly wasted.
- Route Google-ecosystem step data through Fitbit. Documented abandon so a future contributor doesn't re-litigate.

### Apple Health (steps / sleep / activity)
- **Feasibility:** SPECULATIVE / impossible on Windows. **No Apple Health Windows API.** HealthKit is Apple-platform-only with no public cloud endpoint. The only access is a manual `export.zip` (`export.xml`) the user drops where Pulse can read it.
- **Value:** iPhone users have the richest archive, but the only delivery is a stale, manual file import.
- **Effort:** days for an export parser; impossible for anything live.
- Be blunt: Pulse cannot talk to Apple Health. The realistic feature is "point Pulse at `export.xml`, parse `HKQuantityTypeIdentifierStepCount`, show last-N-days." Parse on the worker; local by construction.

### Garmin Connect (unofficial)
- **Feasibility:** SPECULATIVE — the **official Developer Program is business-use only**. Unofficial: `python-garminconnect` (active; its former auth dep `garth` is deprecated), undocumented endpoints Garmin can block, storing the user's *account credentials* (not a scoped token) is a liability.
- **Value:** Garmin owners are numerous/data-rich, but the path is fragile and credential-hungry.
- **Effort:** days to wire; ongoing weeks of breakage.
- A wildcard at best. Never store the Garmin password in plaintext — DPAPI-wrap (`win32crypt.CryptProtectData`). Poll daily, sparingly. Local-only.

### Oura API (sleep / activity / readiness)
- **Feasibility:** Verified, fresh gotcha. v2 endpoints `/v2/usercollection/daily_sleep`, `/daily_activity`, `/daily_readiness`. **PATs were deprecated Dec 2025 and no longer work** — **OAuth2 only** now.
- **Value:** Oura's sleep staging + readiness score is exactly the two-number morning glance a tray companion shows well.
- **Effort:** days. Same OAuth2 PKCE/localhost machinery as Fitbit; clean data shapes.
- The PAT deprecation is the trap a 2024 tutorial leads you into. Daily endpoints update once/day; poll on dashboard-open + low-frequency timer. Persist only the refresh token + last day's summary.

### Whoop API (sleep / recovery / strain)
- **Feasibility:** Verified — v2 endpoints (sleeps, recoveries, cycles, workouts), OAuth2 (authorization code; `offline` scope for refresh). Access tokens ~hourly; ~100 req/min, 10,000/day.
- **Value:** Whoop's Recovery Score (0–100) is a flagship one-glance metric ideal for a tray badge; regenerates each morning.
- **Effort:** days. OAuth2 + hourly token refresh on the worker.
- Refresh proactively (or lazily on 401) from the worker, never blocking the GUI. One fetch on first dashboard-open of the day suffices. DPAPI-wrap the refresh token.

### Water intake (manual logging + persistence)
- **Feasibility:** Verified — pure local, no API. A counter widget with `+250 ml`/`+1 glass` buttons, a daily goal, midnight reset; persist to JSON (`utf-8-sig` reads).
- **Value:** The most-requested QS feature that needs zero integrations. A tray app that's already open is the lowest-friction place to tap "+1 glass."
- **Effort:** hours.
- Wire `clicked` directly (the dashboard never holds focus — don't rely on focus events). Roll over by comparing stored date to `date.today()` on open. The privacy-first archetype.

### Sleep last night (Fitbit / Oura / Whoop)
- **Feasibility:** Verified, as a presentation layer over the three wearable integrations. Fitbit `/1.2/user/-/sleep/date/{date}.json`; Oura `/daily_sleep`; Whoop sleeps endpoint. All read-only over the same OAuth tokens.
- **Value:** "How did I sleep?" (duration + score) is arguably the highest-value daily glance a tray companion offers — the metric people actually check each morning.
- **Effort:** hours **once any one** wearable OAuth is wired; otherwise inherits that integration's days.
- Don't build a separate auth path — consume whichever wearable the user connected. Normalize the three providers' fields into one internal shape. Fetch on first dashboard-open of the day.

### Screen time — local active-window tracking (Win32)
- **Feasibility:** Verified — `GetForegroundWindow` → `GetWindowThreadProcessId` → process name (Pulse already uses the pattern). Sample 1–5s on the worker, bucket dwell time per exe.
- **Value:** A self-contained, no-cloud screen-time breakdown. No accounts, no OAuth.
- **Effort:** days.
- The privacy-first ideal. Store process-level aggregates by default; title capture opt-in. Idle detection via `GetLastInputInfo`.

### Screen time — ActivityWatch (local REST API)
- **Feasibility:** Verified — open-source, local, REST at `http://localhost:5600/api/` (`/api/0/buckets`, `/buckets/{id}/events`). No auth (localhost-only).
- **Value:** If the user already runs AW, Pulse surfaces rich, pre-categorized data (AFK, browser-tab watchers) without re-implementing tracking.
- **Effort:** hours (read events) / days (query-API aggregations).
- Probe `localhost:5600` on the worker; degrade gracefully if absent. Privacy posture unchanged — nothing leaves the box.

### Screen time — RescueTime API
- **Feasibility:** Verified, cloud-dependent. **Analytic Data API** (`/anapi/data`, API key) + **Daily Summary Feed** (`/anapi/daily_summary_feed`, prior-day rollups).
- **Value:** "Productive vs distracting" + category breakdowns Pulse can't compute itself. The Daily Summary is purpose-built for a "show me yesterday" notification.
- **Effort:** hours — one API key (no OAuth) + one GET.
- The honest caveat: RescueTime is third-party cloud, so this sends activity to their servers (the user already opted into that by running it). DPAPI-wrap the key.

### Keystrokes / clicks per day (global input counting)
- **Feasibility:** Verified but fraught — `WH_KEYBOARD_LL`/`WH_MOUSE_LL` via `SetWindowsHookEx` (or `pynput`). **Count only — never key identity or text.**
- **Value:** A "12,400 keystrokes / 3,100 clicks today" novelty for the QS-curious.
- **Effort:** days (raw ctypes) / hours (`pynput`).
- **The AV/privacy landmine.** A global low-level keyboard hook is structurally a keylogger — AV/EDR flags the unsigned `.exe`. Mitigations: store only integer counts; explicit opt-in with an in-app explanation; local-only; warn users their AV may complain. Hook proc does nothing but `count += 1` and return `CallNextHookEx`.

### Scroll distance (WH_MOUSE_LL wheel deltas)
- **Feasibility:** Verified — on `WM_MOUSEWHEEL`, read the high word of `mouseData` in `MSLLHOOKSTRUCT` (signed × `WHEEL_DELTA` 120) and accumulate.
- **Value:** A whimsical "doomscroll-o-meter" companion to the click counter.
- **Effort:** hours once the mouse hook exists; days standalone.
- Inherits the exact AV-adjacency and privacy posture of the input-hook feature — same opt-in/counts-only/local-only. Wheel deltas carry no content.

### Typing speed / WPM
- **Feasibility:** SPECULATIVE (derivation) — `(chars/5)/minutes` over typing bursts from the keyboard counter. Rough, since without key identity you can't distinguish words from backspaces.
- **Value:** "Avg 68 WPM during focused work" — satisfying for writers/coders, but soft accuracy.
- **Effort:** days — burst detection (gap segmentation) on the hook counter.
- Label it an *estimate*. Same opt-in/local-only/counts-only constraints — no text stored, which is precisely why it's approximate.

### Mouse travel distance (pixels)
- **Feasibility:** Verified — `WH_MOUSE_LL` cursor `pt` deltas, or sample `GetCursorPos` on a timer (lighter).
- **Value:** "Your cursor traveled 1.2 km today" — pairs with scroll distance.
- **Effort:** hours.
- The **`GetCursorPos`-polling variant needs no `SetWindowsHookEx`** — it avoids the keylogger-adjacency AV flag entirely; a 50–100ms poll is accurate enough. Store only the accumulated total. The smart move.

### Standing / sitting reminders
- **Feasibility:** Verified — pure local `QTimer`; tray notification ("Stand up — sitting 50 min"), suppress while idle (`GetLastInputInfo`).
- **Value:** Broad ergonomic appeal; dovetails with the planned notifications.
- **Effort:** hours.
- The lowest-risk feature here: no API, no hook, no stored telemetry beyond an interval.

### Blink / break reminders (20-20-20)
- **Feasibility:** Verified — same `QTimer` pattern; every 20 min prompt "Look 20 feet away for 20 seconds," optionally tied to active screen time.
- **Value:** Eye-strain relief for anyone at a screen all day — a caring micro-feature.
- **Effort:** hours.
- Shares one reminder subsystem with standing reminders. If wired to active-window/idle data, fires only during real screen use.

**Greenlight candidates:** Water intake (zero integrations, the canonical privacy-first feature); standing + blink/20-20-20 reminders (no telemetry, slots into planned notifications); local active-window screen time (Win32 Pulse half-owns); mouse travel via `GetCursorPos` (fun metric, no global hook, dodges AV risk); sleep + steps via one wearable OAuth (Oura *or* Fitbit *or* Whoop).
**Wildcard candidates:** ActivityWatch ingestion (excellent if AW is running; hard dependency); RescueTime daily summary (trivial key, but third-party cloud); keystroke/click/scroll counters + WPM (fun but AV false-positive magnet — opt-in, counts-only, with warning); Garmin via `python-garminconnect` (unofficial, credential storage, breakage); Apple Health export-file import (stale one-time parse, only possible Apple path).
**Abandoned:** Google Fit REST (deprecated, no future); Health Connect (Android-only, unreachable from Windows); Apple Health live/programmatic (no Windows API); official Garmin Developer Program (business-only).

## Power-User Infrastructure & AI Integration

Ten capabilities that turn Pulse from a closed loop into a *platform* — something other software can read from, push to, and extend. The connecting thread is the worker-thread invariant: all I/O and any blocking/async event loop lives off the Qt main thread; anything touching a `QWidget` marshals back via a signal on a `QObject`. Several items introduce a *second* event loop (asyncio for MCP, an HTTP server, a hotkey pump) — none may run on the Qt main thread. The **MCP server is the keystone**: once Pulse speaks MCP, the CLI companion, the webhook story, and the local-LLM summary can all ride the same internal data seam.

### Global Hotkey to Summon the Popup
- **Feasibility:** Verified — `RegisterHotKey` (`user32.dll` via ctypes) delivers `WM_HOTKEY`; or the `keyboard`/`global-hotkeys` libs. **Qt `QShortcut` will NOT work** — focus-scoped, and the dashboard never takes focus. Run a `PeekMessage` pump on a dedicated thread; on hotkey, emit a signal to the main thread which calls `show_near_tray()`.
- **Value:** Power users summoning Pulse dozens of times a day; removes the tray-hunt (relevant given the first-run overflow quirk).
- **Effort:** hours (ctypes) to ~1 day (configurable binding + conflict handling).
- `RegisterHotKey` returns `False` silently if the combo is taken — check it and toast. The `keyboard` lib installs a low-level hook (AV-flagged); prefer raw `RegisterHotKey`.

### MCP Server Exposing Pulse's Data (the keystone)
- **Feasibility:** Verified — official MCP Python SDK (`mcp`, ships `FastMCP`). Transports: **stdio** (zero-config, local) and **Streamable HTTP** (POST + SSE; the modern replacement for deprecated SSE). The SDK runs on `anyio`/asyncio — its own event loop, which **cannot share the Qt main thread**.
- **Value:** The strategic unlock. Claude Code / Claude Desktop can answer "what's my balance and burn rate?", "which pinned models spiked today?", or "tell me when balance drops below $5" — with no chat UI built by Pulse. Pulse becomes the authoritative local context source; the AI does the natural-language layer for free.
- **Effort:** days (read-only server); 1–2 weeks (event subscriptions + polished Claude Desktop config).

A tray app is *already running* — it is not a subprocess a client spawns, which rules out the naïve stdio pattern. Two architectures: **(1) Streamable HTTP, hosted in-process (recommended)** — FastMCP bound to `127.0.0.1:<port>`; clients connect to the running instance. This matches the single-instance-mutex model: exactly one Pulse owns the port and holds live data. **(2) A thin stdio shim** that is itself an HTTP client to the running Pulse — more moving parts, only for clients that can't do HTTP.

Threading: run the MCP server on a dedicated thread, never the Qt main thread:

```python
def _run_mcp():
    mcp.run(transport="streamable-http")  # spins up uvicorn + its own loop
threading.Thread(target=_run_mcp, daemon=True).start()
```

The rule generalizes from AGENTS.md: **MCP tool = read snapshot freely; mutate-then-render = emit a signal to main.** Tool functions never touch `QWidget`s — they read a thread-safe snapshot guarded by a lock; UI-affecting writes emit a Qt signal.

```python
@mcp.tool()
def get_balance() -> dict:
    """Current OpenRouter balance and today's usage."""
    s = snapshot()
    return {"balance_usd": s.balance, "usage_today_usd": s.usage_daily,
            "burn_rate_usd_per_hr": s.burn_rate, "as_of": s.ts}

@mcp.tool()
def set_alert(metric: str, op: str, threshold: float, for_minutes: int = 0) -> str:
    """Register an alert. Mutates state — emits a Qt signal; no widget access."""
    pulse_bus.alert_requested.emit(metric, op, threshold, for_minutes)
    return f"Alert set: {metric} {op} {threshold}"
```

Gotchas: bind `127.0.0.1` only (never `0.0.0.0`). Handle a stale-process port collision gracefully. Document the exact Claude Desktop / `.mcp.json` snippet (where users get stuck). Keep tools read-mostly initially — `set_alert`/`pin_model` cross into "agent can change my app" and deserve a confirmation toast.

### Local LLM for Daily-Summary Text (Ollama)
- **Feasibility:** Verified — `POST http://localhost:11434/api/generate` (or `/api/chat`), `"stream": false` for a single JSON object. Plain `requests.post` on the worker; text delivered via signal.
- **Value:** Turns the planned daily-summary toast from a templated string into prose. Privacy-positive; the rare AI feature that *reduces* OpenRouter spend.
- **Effort:** hours.
- Assemble stats deterministically, ask the LLM only to *phrase* them. Catch `ConnectionError`/404 and fall back to the template so the notification always fires. Never let the LLM invent numbers. Model configurable (`summary_model`).

### Webhook Receiver (Embedded HTTP Server)
- **Feasibility:** Verified — `http.server` (stdlib) or Flask/FastAPI, on a dedicated thread. Handlers parse + emit a signal, never touch widgets. If you adopt the MCP Streamable-HTTP server, **co-host the webhook on the same ASGI app/port**.
- **Value:** External services *push* to Pulse — CI finishes → toast; a budget service crosses a threshold → alert. Inverts Pulse from passive monitor to event sink.
- **Effort:** hours (stdlib, one route) to ~1 day (auth + event types).
- **Bind localhost-only and require a shared-secret token** even on loopback (any local process could inject fake toasts). Validate-and-signal only; don't do real work in the handler thread.

### CLI Companion (`pulse balance`, `pulse pin gpt-5`)
- **Feasibility:** Verified — `argparse`/`click`/`typer` entry point; a separate short-lived process talking to the running instance via (a) HTTP to the MCP/webhook server (**preferred**), (b) a Windows named pipe, or (c) a localhost TCP socket.
- **Value:** Scripters and terminal-dwellers. `pulse balance --json | jq`; `pulse pin gpt-5` in a dotfile.
- **Effort:** days — mostly the protocol + packaging the `console_scripts` entry point.
- Don't invent a third IPC mechanism — if the MCP HTTP server exists, the CLI is just an HTTP/MCP client hitting the same tools. Handle "Pulse isn't running" cleanly; respect the single-instance mutex (don't spawn a duplicate).

### Scripting for Custom Widgets (Python plugins / Lua via lupa)
- **Feasibility:** Verified loading (`importlib`/`spec_from_file_location`; Lua via `lupa`). **SPECULATIVE on safety:** neither sandboxes. `importlib`-loaded Python runs with full process privileges (can read `settings.json`, exfiltrate the key). Lua via lupa is narrower but not a boundary.
- **Value:** Tinkerers and the long tail of "I wish Pulse showed X."
- **Effort:** weeks to do *responsibly* (loading is easy; a trust model + API surface + not breaking dashboard invariants is hard).
- A Python plugin is **arbitrary code execution by design** — fine for *local user-authored* scripts, unacceptable for *shared/downloaded* ones. For shareable widgets, Lua-via-lupa with a deliberately tiny API (no `os`/`io`) is safer. A script must never call into Qt off the main thread — wrap the plugin API so widget creation is always marshaled.

### Plugin Architecture (entry points / auto-discovered folder)
- **Feasibility:** Verified — (1) `importlib.metadata.entry_points(group="pulse.plugins")` for pip-installed plugins; (2) a `%APPDATA%/Pulse/plugins/` folder scanned with `importlib`. **Entry points are unreliable under PyInstaller** (frozen builds don't expose third-party metadata), so **folder-scan is the safer primary for the shipped `.exe`**.
- **Value:** The formalization that makes everything coherent — exports, custom widgets, alert handlers, and MCP tools all become plugins with a common lifecycle. What makes Pulse "a platform."
- **Effort:** days for a clean manager + reference plugin.
- Design around the existing sample stream: `on_sample(snapshot)` (worker thread, so plugins can do I/O) + a `ui` handle whose methods emit signals. Version the API (`PLUGIN_API = 1`); wrap callbacks in try/except and disable throwing plugins — one bad plugin must not crash the tray.

### Exports: CSV, Prometheus, Notion, JSON
- **Feasibility:** Verified. **JSON/CSV** stdlib over persisted history. **Prometheus** `prometheus_client` — `start_http_server(port)` launches a daemon-thread exporter; set `Gauge('pulse_balance_usd', …).set(value)` from the worker; Grafana scrapes `/metrics`. **Notion** `notion-client` or raw `POST /v1/pages`.
- **Value:** **Prometheus is the standout** — drops Pulse into Grafana for free balance/burn/latency time-series, exactly the historical view the dashboard can't show. CSV/JSON for backups; Notion for the "log my life in Notion" crowd.
- **Effort:** CSV/JSON hours; Prometheus hours–1 day; Notion ~1 day.
- Prometheus: if you've adopted the MCP/webhook ASGI app, expose `/metrics` there via `make_asgi_app()` to avoid another port. Set gauges from the worker's sample callback. Notion: the database must be explicitly shared with the integration (users forget); ~3 req/s, batch daily.

### Alert Rules Engine (declarative rules over the sample stream)
- **Feasibility:** Verified — pure Python; runs inside the worker's existing sample loop, emits a signal when a rule fires. "For 5 minutes" / "after 5pm" are stateful predicates over the timestamped stream you already buffer.
- **Value:** The engine behind every notification users want, and the MCP `set_alert` tool + webhook events register rules through the same path. Connective tissue between "Pulse has data" and "Pulse tells me things."
- **Effort:** days (robust evaluator with duration/hysteresis); hours (threshold-only).
- Rules as data: `{"metric":"balance","op":"<","threshold":5,"for_minutes":0,"and":{"after_hour":17}}`. **Debounce/hysteresis** — fire on the *transition* into the alert state with a clear-band, don't re-fire until it clears. Keep the grammar tiny. Emit `alert_fired(rule_id, message)` to main; never toast from the worker.

### Conditional Notifications (Windows Toast)
- **Feasibility:** Verified — `win11toast` (rich: images, audio, buttons, callbacks) and `Windows-Toasts` (clean, maintained); `plyer` as a weak cross-platform fallback. Raise the toast from the **main thread** on an `alert_fired`/`event_received` signal.
- **Value:** The user-facing payoff for the whole alert/webhook/summary stack — roadmap #1. Native toasts respect Focus Assist and land in the Action Center.
- **Effort:** hours (fire-and-forget); ~1 day (interactive buttons → open dashboard, snooze, deep-link to top-up).
- Toasts attribute to an **AppUserModelID** — register one or they show as generic "Python." WinRT callbacks fire on a WinRT thread — marshal back via signal. Provide a per-rule mute (a toast storm gets you uninstalled).

**Greenlight candidates:** MCP server (Streamable HTTP, in-process worker thread — build first); alert rules engine + Windows toasts (roadmap #1, pure-Python/WinRT); local-LLM daily summary (Ollama, template fallback); Prometheus exporter (free Grafana graphing); global hotkey (`RegisterHotKey`).
**Wildcard candidates:** CLI companion (cheap *after* MCP); webhook receiver (co-host on the MCP ASGI app, localhost + token mandatory); plugin architecture (folder-scan over entry points; formalize once 2–3 features exist to refactor into it); Notion export (niche, most API friction).
**Abandoned:** Qt `QShortcut` for the global hotkey (focus-scoped, never fires); naïve stdio-spawn MCP (conflicts with the single-instance live-data model); any second event loop on the Qt main thread (blocks `app.exec()`); `importlib`/`lupa` as a security sandbox for *shared/downloaded* scripts (neither is a trust boundary).

## Multi-Platform Strategy

Pulse is welded to Windows in exactly four places (confirmed by reading the source): the single-instance kernel mutex (`main.py:66`, `CreateMutexW("Global\\Pulse_SingleInstance_v1")` + `GetLastError()==183`); the no-activate show path applying `WS_EX_TOOLWINDOW` via `GetWindowLongW`/`SetWindowLongW` then `SetWindowPos(HWND_TOPMOST, … SWP_NOACTIVATE)` (`dashboard.py:804-825`); click-outside dismiss polling `GetForegroundWindow` + PID compare (`dashboard.py:845-867`); and startup-on-login via the HKCU `Run` key (`tray_icon.py`). **Everything else is portable Qt** — `widgets.py` imports only PySide6, `dashboard.py`'s UI construction is pure Qt, and `tray_icon.py` already subclasses `QSystemTrayIcon` and paints its gauge with `QPainter`. The port is not a rewrite; it's carving ~60 lines of Win32 glue out of two files behind an interface. macOS is a real shippable target with a known cost ($99/yr + abstraction work); Linux is feasible on X11/KDE and a coin-flip on GNOME/Wayland — because the *platform* fights frameless-popup positioning, not because of Pulse's code.

### Platform abstraction layer (`platform/` package)
- **Feasibility:** Verified — the Win32-bound surface is already isolated to `main.py` (mutex) and `dashboard.py` (`_show_no_activate`, `_check_outside_click`); `widgets.py` and most of `dashboard.py` are pure Qt.
- **Value:** Foundational — nothing else ships without it; also makes the Windows code testable by mocking the adapter.
- **Effort:** days.
- A `TrayAdapter` ABC: `create_tray`, `set_icon`, `set_tooltip`, `position_window`, `single_instance_guard`, `notify`, plus `no_activate_show(widget)` and `register_dismiss_watcher(widget, on_outside)`. The click-outside dismiss is the leakiest abstraction — expose *intent* ("notify me when focus leaves the popup"), not mechanism, so each backend chooses `GetForegroundWindow` polling, `NSApplication` deactivation, or a Qt `eventFilter`.

### macOS via QSystemTrayIcon (recommended)
- **Feasibility:** Verified — Qt docs: QSystemTrayIcon "is supported on all supported versions of macOS," rendering as a menu-bar item; `tray_icon.py` already extends it.
- **Value:** Mac users (a large share of OpenRouter's dev audience) get a native menu-bar gauge with the least code divergence.
- **Effort:** days (within the abstraction work).
- The catch is the icon: macOS menu-bar icons should be *template images* (monochrome, system-tinted). Pulse's colored conic-gradient gauge looks wrong as a flat template — either accept a non-native colored icon (many apps do) or render a monochrome variant. Set `LSUIElement`/`NSApplicationActivationPolicyAccessory` so no Dock icon appears.

### macOS via rumps + embedded Qt popup (not recommended)
- **Feasibility:** SPECULATIVE — rumps is real (wraps `NSStatusBar`), but bridging a rumps-owned status item to a separately-owned Qt popup means reconciling PyObjC's runloop with Qt's event loop.
- **Value:** Marginal — only "more native menu behavior," which QSystemTrayIcon already provides.
- **Effort:** weeks.
- rumps shines for *pure* menu-bar apps; Pulse's value is the rich painted dashboard, which is irreducibly Qt. Two GUI toolkits' run loops is exactly the class of bug AGENTS.md warns about. Use only if QSystemTrayIcon proves unacceptable in testing — verify empirically first.

### macOS popup positioning near the menu bar
- **Feasibility:** Verified — `QApplication.screenAt`/`availableGeometry` account for the menu bar, and `QSystemTrayIcon.geometry()` returns the menu-bar item's rect on Mac (the call `show_near_tray` already relies on at `dashboard.py:774`).
- **Value:** Correct anchoring so the popup drops *below* the menu-bar icon instead of above it (the Windows assumption at `dashboard.py:784`).
- **Effort:** hours.
- The default branch must place the popup *below* the icon rect on macOS. The Win32 `_show_no_activate` is replaced by `WA_ShowWithoutActivating` (already set at `dashboard.py:147`) + `Qt.Tool` + `WA_MacAlwaysShowToolWindow`.

### macOS code signing & notarization
- **Feasibility:** Verified — requires Apple Developer Program ($99/year); Gatekeeper needs a Developer ID-signed, notarized, hardened-runtime build, and PySide6 needs the `com.apple.security.cs.allow-unsigned-executable-memory` entitlement.
- **Value:** Without it, every user must right-click → Open past a scary warning. For a tool holding an API key, a clean install matters.
- **Effort:** days (first time), then hours per release once scripted.
- The real recurring macOS cost. The hardened runtime breaks Python's executable-memory use without the entitlement; known PyInstaller notarization quirks (`--onedir` vs `--onefile`). Keep PyInstaller (one packaging mental model with Windows); wrap `codesign` + `notarytool` + `stapler` in a script.

### Linux tray via QSystemTrayIcon + StatusNotifierItem
- **Feasibility:** Verified — Qt docs: works on Linux DEs implementing D-Bus StatusNotifierItem (KDE, GNOME, Xfce, LXQt, DDE) and X11 environments implementing XEmbed.
- **Value:** KDE/Plasma, Xfce, LXQt, Cinnamon users get a working tray gauge essentially free once the adapter exists.
- **Effort:** days.
- Qt talks D-Bus `org.kde.StatusNotifierItem` directly — no manual `libayatana-appindicator` binding needed (that's for GTK apps). Ships `libxcb*` + Qt platform plugins. Test KDE (best) and Xfce (XEmbed) before claiming support.

### Linux on GNOME (requires user-installed extension)
- **Feasibility:** Verified — GNOME Shell dropped legacy tray support; an SNI tray needs the user to install the AppIndicator extension, and under Wayland to log out/in after.
- **Value:** GNOME is the most common default; ignoring it loses most Linux desktop users — but the fix is on the user's side.
- **Effort:** hours (code) + documentation.
- No code change makes Pulse appear in a stock GNOME shell. Detect-and-guide: check whether an SNI watcher is registered on D-Bus; if not, surface a one-time "Install the AppIndicator extension" message with a link (mirrors the honesty about the Windows overflow-tray quirk). Even with the extension, GNOME re-colors/re-sizes icons.

### Linux frameless-popup positioning under Wayland
- **Feasibility:** SPECULATIVE / known-broken — as of a Nov 2025 survey and the Qt Wayland issues wiki, Wayland refuses client-driven absolute positioning by design; `move()`/`setGeometry()` on toplevels are silently ignored.
- **Value:** The single biggest threat to a *good* Linux experience — the dashboard could pop up dead-center instead of anchored to the tray.
- **Effort:** weeks (and may be unsolvable cleanly).
- The clean Wayland answer (`xdg-popup` anchored to the tray surface) isn't reachable: Qt's `QSystemTrayIcon` exposes no "show popup anchored to this tray item" API. Practical options: (1) accept compositor-chosen placement on Wayland and skip anchoring; (2) recommend the X11/XWayland session where `move()` works; (3) render the dashboard inside the tray menu (a major rewrite). Ship option 1 with a documented caveat; don't pretend it's solved.

### Linux single-instance via flock / D-Bus
- **Feasibility:** Verified — `fcntl.flock` on a lockfile in `$XDG_RUNTIME_DIR`; or a D-Bus well-known name.
- **Value:** Replaces the Windows named mutex so a second launch focuses the existing instance.
- **Effort:** hours.
- `CreateMutexW` maps cleanly to an exclusive `flock` on `~/.config/Pulse/pulse.lock` (kernel releases it on crash — more robust than a stale PID file). D-Bus is nicer for the "already running, show the dashboard" UX. Start with flock; macOS can use the same.

### Packaging per platform
- **Feasibility:** Verified — Windows PyInstaller (current); macOS PyInstaller→.app→.dmg or py2app; Linux AppImage / `.deb` / Flatpak.
- **Value:** Each OS expects its own install vehicle; mismatched packaging is the difference between "ran it" and "gave up."
- **Effort:** Windows done; macOS days; Linux days per format.
- **Windows** stays as-is (the `sys.stderr = None` redirect is already handled by `_redirect_streams_if_frozen`). **macOS**: PyInstaller → `.dmg` → the signing pipeline above. **Linux**: AppImage is the lowest-friction first target; `.deb` needs Qt-plugin dependency declarations; Flatpak's sandbox complicates D-Bus tray registration and autostart — defer past AppImage. Autostart diverges per OS (Windows `Run` key, macOS `LaunchAgent` plist, Linux XDG `.desktop`) — the `TrayAdapter` should own this too.

**Greenlight candidates:** the `platform/` abstraction layer (`TrayAdapter`); macOS via `QSystemTrayIcon`; macOS popup positioning fix (anchor below the menu bar); Linux tray via StatusNotifierItem on KDE/Xfce/LXQt; cross-platform single-instance via `flock`.
**Wildcard candidates:** macOS signing + notarization (necessary but $99/yr + hardened-runtime pain); GNOME support (trivial code, depends on the user installing the extension); Wayland popup positioning (likely can't anchor — accept compositor placement); Linux Flatpak (desirable for Flathub but sandbox complicates tray/autostart).
**Abandoned:** rumps + embedded Qt popup (QSystemTrayIcon already covers the menu-bar item; bridging two run loops is weeks of risk); manual `libayatana-appindicator` bindings (unnecessary — Qt speaks SNI natively); promising a pixel-anchored dashboard on Wayland (architecturally blocked).

# Architecture Notes — toward a plugin / card-stack model

Every claim here is grounded in the actual v0.4.1 code (nine modules, <3,000 lines).

### How the dashboard is laid out today
The dashboard is **already a single vertical stack**, but hand-assembled, not data-driven. `Dashboard._build_ui` (`dashboard.py:178`) roots a `QVBoxLayout` wrapping a `QFrame`, inside which a `GradientStrip` sits above a `QScrollArea` whose content owns `self._content = QVBoxLayout(...)` (`dashboard.py:209`). Sections are appended in fixed sequence: header, error banner, gauge (Credit Balance), usage (TimelineChart + 2 StatCards), burn rate, pinned models, quick links, then `addStretch()`.

Crucially, **the Pinned Models section already implements the card-stack pattern in miniature**: `self._pinned_layout = QVBoxLayout(...)` (`dashboard.py:345`) and `set_tracked_models` (`dashboard.py:367`) dynamically create, remove, and **reorder** `PinnedModelCard` widgets keyed in `self._pinned_cards`, with the reorder loop (`dashboard.py:387-391`) being exactly the "detach all, re-add in wanted order" technique drag-to-reorder needs. The individual cards (`StatCard`, `ArcGauge`, `BurnRateBar`, `TimelineChart`, `PinnedModelCard`) are all `QWidget` subclasses that paint in `paintEvent` and expose a `set_*` data-injection method — already render-only, already `Fixed` height / `Expanding` width. So the refactor target isn't exotic; it's *generalizing the Pinned Models mechanism to the whole dashboard.*

### Worker-thread + signals data flow
`APIWorker(QObject)` (`api_client.py:368`) wraps a synchronous `APIClient` and is `moveToThread(self.api_thread)` (`main.py:96-98`). `FetchTrigger(QObject)` (`main.py:73`) exposes `fetch_key`/`fetch_endpoints(str)`/`fetch_models`, connected to worker `@Slot`s; results return via `key_info_ready(object)`/`endpoints_ready(str,object)`/`models_ready(object)`/`error(str)` connected to controller slots that run on the main thread — **the only place widgets are touched**. Two `QTimer`s drive polling: `key_timer` (60s default) and `endpoints_timer` (300s). **Every card already wants exactly this shape**: a poll trigger on the worker, a result signal back to main.

### Persistence & ctypes layer
`persistence.py` stores a rolling list of `Snapshot` dataclasses in `%APPDATA%/Pulse/state.json` via temp-file-then-`replace` atomic swap, pruned to 90 days / 20k snapshots, read with `encoding="utf-8-sig"` (BOM tolerance), with 5-minute idle dedup. `settings.py` drops unknown keys on load (`settings.py:70-71`) — **so a plugin can add settings keys without breaking the loader, and old configs survive new code.** The mutex (`main.py:65`), `_show_no_activate` (`dashboard.py:804`), and `_check_outside_click` (150ms `GetForegroundWindow` poll, `dashboard.py:845`) are the OS-specific bits; the picker dropdown is an overlay child while the provider popup is a top-level `Tool` window — any new card with a floating element must respect that split.

### The plugin / card-stack design
Introduce a `Card` ABC, a `CardRegistry`, and a `CardHost` owning the `QVBoxLayout`. The lifecycle mirrors the existing worker flow, preserving every threading invariant: (1) the host instantiates each card on the **main thread** (cards are `QWidget`s); (2) a scheduler emits a poll request onto the **worker thread** where `poll()` runs and returns a plain data object (no widget access); (3) the result returns via a single `cardData(card_id, payload)` signal to the main thread, where the host calls `card.render(payload)`. This is `FetchTrigger → APIWorker → *_ready → _on_* → update_*` generalized. The worker stays one `QThread`; cards never spawn their own.

```python
class Card(QWidget):                      # MUST stay a QWidget
    card_id: str
    title: str
    default_interval: int = 60            # seconds; 0 == event-driven
    def poll(self, ctx) -> object: ...    # WORKER thread; no Qt calls
    def render(self, payload) -> None: ...# MAIN thread; safe to update()
    def save_state(self) -> dict: ...

class CardRegistry:
    def discover(self) -> list[type[Card]]:
        cards = list(_BUILTIN_CARDS)                         # phase 1
        for ep in importlib.metadata.entry_points(group="pulse.cards"):
            cards.append(ep.load())                          # phase 2
        for mod in _scan(Path(state_dir()) / "plugins"):     # phase 3
            cards.append(mod.CARD)
        return cards

class CardHost(QWidget):
    def __init__(self, registry, settings, ctx):
        self._layout = QVBoxLayout(self)  # the existing pattern, generalized
        self._order = settings.card_order or [c.card_id for c in ...]
```

**Discovery — phased.** Phase 1: a hardcoded `_BUILTIN_CARDS` list (zero risk, ships the architecture). Phase 2: `entry_points` for pip-installable plugins — but PyInstaller freezes the environment, so these only help dev installs. Phase 3: a `%APPDATA%/Pulse/plugins/*.py` folder scan, which *does* work in the frozen exe (user-writable disk at runtime). **For a single-user tray app, Phase 3 is the highest-value mechanism**; entry points are mostly ceremony.

**Config & reorder.** Extend `Settings` with `card_order`, `disabled_cards`, `card_config` (free-form `{card_id: {...}}`) — additive and backward-compatible since the loader drops unknown keys. Drag-to-reorder via small grip glyphs on a generalized `SectionHeader` calling the existing detach-and-re-add loop (preferred over `QListWidget`, which fights `QScrollArea` nesting + variable card heights). Persist order on drop; keep the open-at-top scroll reset.

**Per-card intervals.** Generalize the two timers to one `QTimer` per distinct interval, grouping cards. `CreditBalanceCard`, `UsageCard`, and `BurnRateCard` all consume the *same* `KeyInfo` poll — cache one fetch per cycle and fan it out rather than triple-fetching `/api/v1/key`.

**Refactor scope.** *Stays untouched:* `widgets.py` paint widgets, `api_client.py`, `persistence.py`, all ctypes window mechanics, the mutex, the tray — **none of the AGENTS.md invariants live in the section-building code.** *Moves:* the seven `_build_*` methods → five `Card` subclasses; explicit signal wiring → a generic `cardData` dispatcher; `_smart_forecast` into the card that owns it. **Estimated effort: ~1 week** (Card ABC + host + scheduler ≈1.5d, migrate 5 sections ≈2d, drag-reorder + settings ≈1.5d, full 20-point re-validation + screenshots ≈1d — mandatory and the riskier-than-it-looks part).

**Preserving validation invariants.** The 20-point list is mostly about the *window* (frameless, no taskbar slot, dismiss, placement) — untouched by the card model. At-risk items: #11 (state persists — covered by additive `Settings` keys + atomic save), #12–#16 (keep `PinnedModelsCard` wrapping the *exact* existing `ModelPicker`/`attach_overlay_to` logic — don't rebuild it), #20 (keep `verticalScrollBar().setValue(0)`). Ship Phase 1 first, validate, then layer in discovery and reordering.

> A caveat worth surfacing: AGENTS.md explicitly lists "add features the user hasn't asked for" and "promise multi-provider abstraction" as *Don'ts*. The card/plugin model is an **architecture investment, not a feature** — pitch it as the thing that *enables* the already-requested roadmap (notifications, cost calculator, per-model spend, settings GUI), not as speculative expansion.

# Composite Product Visions

The same engine — worker-thread polling, frameless popup, card stack, persistence, alert rules, MCP seam — can be aimed at four distinct products. They are not mutually exclusive (the card model lets a user mix them), but naming them clarifies *who* Pulse is for and which greenlights cluster. Each vision is sized as a coherent shippable identity, not a feature dump.

### Vision A — "The AI Spend Cockpit" (the natural extension of today)
Pulse stays a spend monitor but stops being single-source. Cards: OpenRouter (current), Anthropic Admin cost/usage, OpenAI Costs, Cursor team spend, and the Claude Code local-`.jsonl` parse. A unified "today across all my AI tools: $11.40" headline sits above per-source cards, each with its own burn-rate forecast reusing the existing `History`/`burn_rate_per_hour` math. The daily-summary toast (LLM-phrased via Ollama) becomes "Your AI spend today: $11.40, up 18% — mostly Claude Code on the refactor." **Strategic fit: highest** — it's the literal roadmap, just widened, and every card reuses the proven OpenRouter shape. The honest constraint is that three of those five sources require org-scoped Admin keys, so the realistic v1 is OpenRouter + Claude Code (local, no key) + whichever Admin key the user actually holds. This is the vision AGENTS.md's roadmap already points at; everything here is additive to it.

### Vision B — "The Developer Cockpit"
Pulse becomes the always-on status bar for a working developer. Cards: GitHub PRs awaiting review + notifications, GitHub Actions CI status ("main: ✓"), Vercel/Netlify deploy state, Linear assigned-issue count, and the AI-spend cards from Vision A as a secondary strip. A global hotkey (Ctrl+Alt+P) summons it between tasks; the command palette (Ctrl+K) jumps to "open failing CI run" or "pin model X." The alert engine fires on "CI went red on main" and "review requested." This is the vision with the tightest audience match — the person who *built an OpenRouter monitor in PySide6* is exactly this developer — and almost all of it is single-PAT, no-OAuth, hours-each integrations. The MCP server makes it self-reinforcing: Claude Code can ask Pulse "is main green?" before suggesting a deploy.

### Vision C — "The System HUD"
Pulse as an ambient machine-vitals companion, leaning into the tray-resident form factor. Cards: NVIDIA GPU (util/VRAM/temp/power), CPU/RAM, network up/down, battery, microphone/camera in-use indicator. Snap-to-corner mode turns it into a persistent corner overlay; alert rules cover "GPU >90% for 5 min" and "VRAM >95%." This is the most *visually* distinctive Pulse and the most on-brand for the AI/ML crowd (who watch GPU and AI-spend in the same breath), but it's the furthest from the current identity and the snap-to-corner mode fights the dismiss model. Best shipped as an optional card cluster, not a separate app.

### Vision D — "The Quantified-Self Companion"
Pulse as a privacy-first life dashboard: water intake, local active-window screen time, mouse-travel/scroll novelty metrics, sleep + steps via one wearable OAuth, standing/blink reminders, and the heat-map-of-your-day view generalized from spend to activity. Everything local-only by default; the only cloud calls are the wearable the user explicitly connected. This is the warmest, most "companion that lives next to me all day" vision — and the one most at odds with AGENTS.md's "don't add features the user hasn't asked for." It's a real product, but it should be a *deliberate* pivot, not creep. The input-hook metrics (keystrokes/WPM) are the AV landmine that could sink the whole app's reputation, so they stay opt-in with a warning regardless of vision.

**The meta-point:** the card/plugin architecture is what lets these coexist without forking. One user runs Visions A+B (the developer who watches spend); another runs A+C (the ML engineer). The product decision isn't "which vision" — it's "ship the card model so the user picks their own vision," with Vision A as the default-on cluster because it's the existing identity.

# Cross-Cutting Design Concerns

These cut across every category above and deserve a single decision each, made once, before the second data source lands. Getting them right early is cheaper than retrofitting across N cards.

### Secrets storage — the one that can't wait
The moment Pulse holds more than one API key (and several greenlights add bearer tokens, PATs, OAuth refresh tokens), plaintext `settings.json` becomes a liability — it's a single file an unrelated process or a synced backup can read. **Decision: introduce a secrets boundary now.** On Windows, DPAPI via `win32crypt.CryptProtectData` binds a secret to the user account with no password prompt; the `keyring` package abstracts this cross-platform (Credential Manager on Windows, Keychain on macOS, Secret Service on Linux) and keeps the door open for the multi-platform port. Keep non-secret config in `settings.json` (the loader's drop-unknown-keys behavior stays useful) but route every `*_api_key`/`*_token`/`refresh_token` through the secrets layer. This is hours of work now and a painful migration later. Note: the *current* OpenRouter key in plaintext is grandfathered, but new sources should not extend the pattern.

### Rate-limit & quota budgeting
Each source has a different ceiling, and a naive "poll everything every 60s" will blow free tiers (CoinGecko Demo 10k/month dies at 60s; GitHub Search is 30/min; Notion is 3/s; Cost Explorer literally bills per call). **Decision: per-card poll intervals are not optional — they're a correctness requirement.** The card model's per-interval scheduler should treat the interval as a documented contract per source (FX 1–2×/day, weather 15–30 min, spend 60–300s, GitHub via `X-Poll-Interval`/conditional requests). Add a cheap global "I'm on battery / I'm idle" signal that backs off all timers — both saves quota and respects the laptop user. Where a source supports conditional requests (GitHub `If-Modified-Since`) or returns a server-dictated interval, honor it; it's free rate-limit budget.

### Settings schema evolution
`Settings.load` already drops unknown keys, which is the right foundation, but a plugin/card world needs more: a `card_config: dict` namespace per card, a `card_order` list, a `disabled_cards` list, and a schema `version` integer so a future migration can rewrite old shapes. **Decision: version the settings file from the next change.** Additive keys are safe today; the version field is insurance for the day a card's config shape must change. Never require a settings field to exist — every reader defaults gracefully (validation item #11: state persists across restart, including *partial* state).

### The unsigned-exe / antivirus reality
AGENTS.md already documents that the PyInstaller `.exe` trips AV on first run and lands in the overflow tray. Several greenlights/wildcards make this *worse*: global input hooks (keystroke/mouse counters) are structurally keyloggers; a global `RegisterHotKey` pump is benign but a low-level `keyboard`-lib hook is not; an embedded HTTP server opens a listening port. **Decision: treat AV-surface as a feature gate.** Features that install `SetWindowsHookEx` hooks or open listening sockets are opt-in, documented as "your AV may flag this," and never default-on. Longer-term, code-signing the Windows build (an EV or standard cert) is the real fix and pairs with the macOS notarization cost — budget both as "distribution maturity," not per-feature.

### Localhost is an attack surface
The MCP server, webhook receiver, Prometheus exporter, and CLI-over-HTTP all imply a listening port. **Decision: bind `127.0.0.1` only, never `0.0.0.0`, and require a shared-secret token even on loopback** — any local process (or a malicious page via DNS-rebinding-style tricks) can otherwise hit an unauthenticated local endpoint and inject fake toasts or read the user's spend. The single-instance mutex makes the port owner deterministic; handle a stale-process port collision gracefully rather than crashing. Co-host MCP + webhook + `/metrics` on one ASGI app/port to minimize the surface.

### The threading invariant, restated as a checklist
Every new card, source, server, and hook obeys the same rule, and it's worth a one-line audit per addition:
1. **Does it do I/O or block?** → worker thread (or its own dedicated thread for a second event loop). Never the Qt main thread.
2. **Does it touch a `QWidget`?** → main thread only, reached via a `QObject` signal. A plain Python class as a signal receiver corrupts the GUI heap (AGENTS.md).
3. **Does it install an OS callback** (`SetWinEventHook`, `SetWindowsHookEx`, `RegisterDeviceNotification`, WinRT toast callback)? → keep the ctypes trampoline alive in a module-level reference (GC of it crashes the process), make the callback near-instant, and marshal real work back via a signal.
4. **Does it create a floating window?** → overlay-child reparented to the dashboard if it stays inside; top-level `Tool` window if it must extend outside (AGENTS.md's documented split).

If a feature can't answer all four cleanly, it's not ready to build.

# A Suggested Build Sequence

Not a commitment — a dependency-ordered path that front-loads the cheap wins, builds the architectural spine once, and defers the friction-heavy items until the platform underneath them exists. Each phase is independently shippable and independently validatable against the 20-point checklist.

**Phase 0 — Roadmap closeout (days).** Ship the already-requested v0.5 items that need no new architecture: daily spend tracking (hours, zero deps), the notifications/alert-rules engine + native toasts (roadmap #1), and the global hotkey. These are pure wins, prove the alert→toast→main-thread flow end-to-end, and need no card refactor. The cost calculator (also roadmapped) fits here too.

**Phase 1 — The card spine (~1 week).** Execute the Card ABC + CardRegistry + CardHost refactor, Phase-1 discovery only (hardcoded `_BUILTIN_CARDS`, no drag). Migrate the five existing sections onto cards. Add `card_order`/`disabled_cards`/`card_config` + a settings `version`. Add the secrets boundary (DPAPI/`keyring`) while you're touching settings anyway. Re-run the full 20-point validation. This phase changes no user-visible behavior — it's the investment that makes everything after it cheap.

**Phase 2 — The first new sources, lowest-friction first (~1 week).** With the card model live, add the no-OAuth, single-key cards that match the audience: NVIDIA GPU (NVML), core system vitals (psutil), FX-in-home-currency (Frankfurter), GitHub PRs + Actions CI (one PAT), Linear, Todoist. Each is hours. This is where Pulse visibly stops being single-source and the user picks their vision.

**Phase 3 — The keystone (days–2 weeks).** Build the in-process Streamable-HTTP MCP server (read-only tools first), bound to localhost with a token. Then the CLI companion and Prometheus exporter ride the same seam for near-free. The local-LLM (Ollama) daily summary slots in here, reusing the Phase-0 toast path. Now Pulse is a platform Claude Code can query.

**Phase 4 — The Admin-key AI sources (days each).** Anthropic Admin cost/usage, OpenAI Costs, Cursor team spend — each a card mirroring the OpenRouter client. Gated on the user actually holding the relevant org key; the UI must degrade honestly ("Admin key required") for users who don't.

**Phase 5 — Polish, breadth, and the optional pivots (ongoing).** Drag-to-reorder + folder-scan plugin discovery; themeable skins; the heat-map and rewind views; the quantified-self cluster (water, screen time, reminders, one wearable OAuth) as a deliberate opt-in; multi-platform abstraction if there's macOS/Linux demand. The wildcards live here, shipped behind flags with eyes open about their risks.

The throughline: **Phase 0 and 1 unlock everything else, and neither requires a single new external API.** A solo dev with an AI pair could plausibly reach the end of Phase 2 — a genuinely multi-source Pulse — in three to four focused weeks, with each week ending on a validated, shippable build.

- **Slack global unread count.** No supported public method; the desktop badge uses the undocumented `client.counts` with an `xoxc` session token. *Trigger:* Slack shipping a supported per-user unread/mention count method.
- **Wayland tray-anchored popup positioning.** Wayland refuses client-driven absolute positioning and `QSystemTrayIcon` exposes no anchored-popup API (still true Nov 2025). *Trigger:* Qt exposing an `xdg-popup`-anchored-to-tray API, or a compositor consensus on tray positioning.
- **GNOME native tray.** Still requires a user-installed AppIndicator extension. *Trigger:* GNOME Shell restoring first-class tray support (no sign of it).
- **Cross-platform fitness data (non-Fitbit).** Google Fit REST is dead, Health Connect is Android-on-device-only, Apple Health has no Windows/cloud API. *Trigger:* any of these (or a new aggregator) shipping a desktop-reachable REST surface. Until then, Fitbit/Oura/Whoop are the only live wearable paths.

# Abandoned

Considered and rejected, with reasons — so the next agent doesn't re-tread this ground. (Distinct from *Watching*: these are rejected on their merits or fit, not merely waiting on a platform. A few overlap where the platform gap is also a fit problem.)

**AI / dev tools**
- **AWS Bedrock spend.** Cost Explorer is **billed per API call ($0.01)** and needs IAM access-key/secret creds — wrong cost model *and* wrong UX for a paste-one-key poll-on-timer tray app. Enterprise-niche.
- **OpenAI legacy `/v1/usage` & `/v1/dashboard/billing/*`.** Dead for normal `sk-` keys (empty arrays / 404); only ever worked with browser session tokens. Never build on them — use the Admin Costs API.
- **Groq / Together / Fireworks cards.** No *verified* public spend/usage read API as of mid-2026; shipping a card would be vapor. (See Watching for the re-trigger.)

**System / hardware**
- **AMD / Intel GPU stats.** AMD's ADLX needs a native binding compiled against the driver SDK (PyInstaller pain); Intel Arc has **no Python NVML equivalent on Windows**. Coverage would be asymmetric and partly impossible — promising "GPU stats" universally would be dishonest. (NVIDIA via NVML is greenlit.)
- **CPU/GPU temps & fan RPM.** Requires **LibreHardwareMonitor running, typically elevated**, exposing `root\LibreHardwareMonitor`. A separate elevated background app is antithetical to a self-contained, unsigned tray companion; dark for nearly all users.
- **Disk SMART health.** The WMI `PredictFailure` bool is unreliable/uninformative; real SMART needs `smartctl` bundled **plus admin**. Elevation + external binary clashes with the no-elevation shipping model; low actionability.
- **Bluetooth connected-device state.** `bleak` is built for scan/connect, not "what's connected now"; reliable connected-state across Classic + LE is finicky and the value is low for this product.

**Productivity**
- **Slack unread badge.** No supported global-count API; the only path is the undocumented `xoxc` session-token hack (fragile, ToS-adjacent). (Watching for a real method.)
- **Discord personal unread.** Requires a self-bot user token, which **violates Discord ToS and risks account ban**. The one integration with account-ban risk — drop entirely.
- **Things task list.** macOS/iOS only, AppleScript/URL-scheme automation, no Windows client or web API — impossible on Pulse's platform.
- **True Focus Assist / Windows 11 DND profile toggle.** Microsoft removed the supported registry/`SystemParametersInfo` path at Win10 1803; the real profile needs undocumented WNF APIs that break without notice. The audio-mute + global-toast-suppress combo delivers the felt benefit without the rabbit hole.

**Finance**
- **Plaid bank balance/transactions.** Plaid Link is a browser/OAuth-redirect flow that can't live in the BypassWindowManager popup and has no desktop callback; live balance is **billed per call**, breaking the poll-on-timer model; and net-worth tracking is a *different product* from an OpenRouter credit watcher with thin audience overlap.
- **exchangerate.host.** Removed its keyless free access (now key-gated) — superseded by Frankfurter / open.er-api.com.
- **Alpha Vantage.** Free tier gutted to **25 requests/day** — effectively unusable for live tickers. (Use Finnhub.)
- **Yahoo Finance unofficial endpoints.** No key but unstable (frequent 429 IP blocks, undocumented breaking changes) — unsafe to ship.
- **Nord Pool / US-utility electricity (general).** Nord Pool's official feed is commercial; the US has no free national API. Only the UK Octopus path is genuinely free/keyless (a regional wildcard).

**Life telemetry**
- **Google Fit REST.** Deprecated (sign-ups closed May 2024, winding down through 2026). Route Google-ecosystem step data through Fitbit instead.
- **Health Connect.** Android-only on-device API, no REST/cloud surface — unreachable from Windows.
- **Apple Health live/programmatic.** No Windows API; HealthKit is Apple-platform-only with no public cloud endpoint. Only a stale manual `export.xml` import is even possible (a wildcard).
- **Official Garmin Connect Developer Program.** Business-use-only — not available to an individual hobby app, so the only Garmin route is the unofficial-library wildcard.

**Infra / AI integration**
- **Qt `QShortcut`/`QKeySequence` for the global hotkey.** Focus-scoped only; the dashboard uses `BypassWindowManagerHint` and never takes focus, so it would never fire. Use Win32 `RegisterHotKey`.
- **Naïve stdio-spawn MCP server.** Assumes the client launches Pulse as a subprocess, conflicting with the single-instance tray model and the live-data requirement. Use in-process Streamable HTTP (keep an optional stdio→HTTP shim only if a client truly can't do HTTP).
- **Any second event loop on the Qt main thread** (FastMCP/uvicorn/FastAPI, or a `GetMessage` hotkey pump). Categorically abandoned — it blocks `app.exec()` and violates the worker-thread invariant. Every such loop gets its own thread with results marshaled via `QObject` signals.
- **`importlib`/`lupa` as a security sandbox for *shared/downloaded* scripts.** Neither is a real trust boundary. Local user-authored scripts are fine (their machine, their key); a marketplace of downloadable Python widgets is abandoned until a genuine isolation story exists. Lua-with-a-tiny-API is the least-bad path if shareable widgets become a goal.
- **`importlib.metadata` entry points as the *primary* plugin discovery.** Doesn't survive PyInstaller's frozen environment — only serves dev installs. Use the `%APPDATA%/Pulse/plugins/*.py` folder scan; keep entry points as an optional dev convenience.

**Multi-platform**
- **rumps + embedded Qt popup (macOS).** `QSystemTrayIcon` already covers the menu-bar item; bridging rumps' PyObjC runloop to Qt's event loop is weeks of risk for marginal native-feel gain, and the dashboard is irreducibly Qt.
- **Manual `libayatana-appindicator` bindings (Linux).** Unnecessary for a Qt app — Qt speaks the StatusNotifierItem D-Bus protocol natively. That library is a GTK-world solution.
- **Pixel-anchored dashboard on Wayland.** Architecturally blocked by Wayland's no-client-positioning stance and Qt's lack of a tray-anchored-popup API. Accept compositor placement with a documented caveat (see Watching).

**UI**
- **Tray-resident mini-terminal.** Off-mission, needs heavy ConPTY/`pywinpty`, collides with the frozen-exe `sys.stderr = None` constraint. Build the roadmapped cost calculator instead.
- **(From the existing roadmap's "Not doing," reaffirmed):** full 340-model grid browser (openrouter.ai/models does it better; the dynamic picker covers find-and-pin); generic 3-dot service-health summary (useless without per-provider granularity); "updated 35s ago" label (nobody watches the clock); any web view / Electron (native PySide6 stays).

For macOS, confirm willingness to pay the $99/yr Apple Developer fee (notarization is otherwise a per-launch scary-warning UX). # Methodology & Confidence

How to read this document, and how much to trust it:

- **The fan-out.** Eight research agents ran in parallel, one per axis (AI/dev tools, system/hardware, productivity, finance, life telemetry, power-user infra, multi-platform, architecture). Each read AGENTS.md and the relevant source, then verified endpoints, library names, WMI namespaces, registry paths, and rate limits against current (mid-2026) documentation via live web search rather than memory. The architecture and multi-platform agents additionally grounded every claim in the actual `main.py`/`dashboard.py`/`widgets.py`/`persistence.py` code, citing `file:line`.
- **What "Verified" means and doesn't.** Verified = the agent confirmed the endpoint/call/library exists and behaves as described, against a primary source, in 2026. It does **not** mean it was end-to-end tested inside Pulse, and it does **not** freeze in time — APIs change, free tiers get cut (exchangerate.host and Alpha Vantage both regressed recently), and "Verified-but-gated" items still need a scope the user may not hold. Re-verify any endpoint immediately before building on it. **SPECULATIVE** is an explicit research task with a named thing to check (see Appendix C), not a soft yes.
- **Effort estimates** assume one solo developer with an AI pair and are deliberately coarse (hours / days / weeks). They fold in the non-obvious costs the research surfaced — OAuth consent screens, PyInstaller bundling of native deps, the mandatory 20-point re-validation after any UI change, ctypes struct plumbing — not just the happy-path code. Treat them as relative sizing, not commitments.
- **A transparency note:** during this run the safety classifier that normally reviews sub-agent output was intermittently unavailable, so several sub-agent results carried an automated "verify before acting" flag. The synthesis above was assembled with that in mind — the higher-confidence claims are the ones tied to a citable primary source or a `file:line` in the repo, and those are what the Top 15 leans on. The further an idea sits toward the Wildcards/Watching end, the more its specifics deserve a fresh check.
- **Scope honesty:** this is an *unbounded* exploration by request — it deliberately ranges past what AGENTS.md sanctions ("don't add features the user hasn't asked for"). The ranking compensates by weighting strategic fit, so the only items that reach "Greenlight Now" are roadmap-aligned or low-risk, on-identity expansions. The genuinely speculative material is quarantined, by design, into Wildcards, Watching, and Abandoned so it informs without misleading.
